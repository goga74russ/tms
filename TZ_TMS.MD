Версия: 3.0 | Дата: март 2026 | Статус: черновик
Профиль: транспортная компания с собственным автопарком
 
Содержание


 
1. Общее описание системы
1.1. Назначение
Операционная платформа для транспортной компании с собственным автопарком. Полный цикл: от приёма заявки до подтверждения доставки. Предрейсовые осмотры (технический + медицинский), формирование путевых листов, контроль рейсов, управление автопарком и ремонтами, контроль РТО по тахографу, учёт пропусков и весовых ограничений, ЭПД, учёт штрафов, тарификация и счета, KPI. Каждая роль получает собственный дашборд.
1.2. Ключевые цели
•	Заменить «1С + Excel + WhatsApp» единой платформой с дашбордами для каждой роли.
•	Электронные журналы предрейсовых осмотров: техосмотр (механик) и медосмотр (медик) с чек-листами и ЭП.
•	Автоматическое формирование путевого листа из данных рейса + техосмотра + медосмотра.
•	Контроль рейсов в реальном времени: карта, статусы, отклонения.
•	Контроль РТО водителей по данным тахографа (Приказ Минтранса №424).
•	Управление ремонтами и ТО: заявки, статусы, затраты, история.
•	KPI руководителя: выручка, себестоимость, маржа, утилизация, простои.
•	ЭПД (ЭТрН, электронные путевые листы) — готовность к обязательному переходу с сентября 2026.
•	Тарификация и выставление счетов/актов клиентам.
•	Учёт пропусков (МКАД/ТТК/зоны), весовых ограничений, штрафов ГИБДД.
•	Оффлайн для водителей. Выгрузка в 1С без жёсткой привязки.
•	Уведомления клиентам о статусе доставки (SMS/push/email).
1.3. Границы системы
1.3.1. Что входит
•	Приём и управление заявками
•	Планирование и исполнение рейсов
•	Предрейсовый техосмотр ТС (электронный журнал)
•	Предрейсовый медосмотр водителей (электронный журнал)
•	Формирование путевого листа
•	Подтверждение доставки, возвраты, претензии
•	Управление автопарком (ТО, ремонты, страховки, пропуска, тахографы)
•	Управление водителями (допуски, РТО, медсправки)
•	ЭПД: ЭТрН, электронный путевой лист, УПД, акты
•	Тарификация и выставление счетов/актов
•	Учёт штрафов ГИБДД
•	Контроль весовых ограничений и пропусков
•	Управленческий учёт, KPI, аналитика
•	Оффлайн-режим
•	Уведомления клиентам
1.3.2. Что НЕ входит
•	Бухучёт, кадры, зарплата
•	Складские операции
•	Прямое управление бортовыми системами (через интеграцию)
•	Контроль холодовой цепи (опциональный модуль в будущем)
1.4. Принципы
1.4.1. Дашборд для каждой роли
Каждый видит ровно то, что нужно ему. Интерфейсы простые и визуальные.
1.4.2. Событийная архитектура
Все действия — неизменяемые события. Отмена — новое событие.
1.4.3. Оффлайн как норма
Мобильные рабочие места работают без интернета. Синхронизация при связи.
1.4.4. Документы как код
При создании рейса — автогенерация плана документов: путевой лист, ТТН, ЭТрН, акты.
1.4.5. Готовность к ЭПД
С сентября 2026 переход на ЭПД обязателен. Система проектируется с поддержкой ЭПД.
1.4.6. Автономность от 1С
Полностью работает без 1С. Коннектор — опционально.
 
2. Роли и дашборды

2.2. Дашборд логиста
•	Канбан заявок: Новая → Подтверждена → Назначена → В пути → Доставлена / Проблема
•	Фильтры по клиенту, дате, маршруту, грузу
•	Индикаторы SLA
•	Быстрое создание из шаблона (постоянные клиенты/маршруты)
•	Реестр документов по заявке: ТТН, ЭТрН, акты, подтверждение
2.3. Дашборд диспетчера
•	Карта: ТС с цветовой индикацией (зелёный/жёлтый/красный/серый)
•	Панель назначения: заявки слева, свободные ТС справа, drag-and-drop
•	Таймлайн занятости ТС
•	Контроль РТО: остаток часов по каждому водителю (тахограф)
•	Проверка пропусков при назначении через зоны ограничения
•	Проверка веса: груз vs грузоподъёмность
•	Быстрая замена ТС/водителя при недопуске
2.4. Дашборд руководителя — KPI
•	Выручка за период, сравнение
•	Себестоимость (топливо, ЗП, ТО, штрафы, Платон)
•	Маржинальность по клиентам/маршрутам/ТС
•	Утилизация парка: % в работе, простой, холостой пробег
•	% доставок вовремя
•	Дебиторская задолженность
•	Затраты на ТО/ремонт: тренд
•	Расход топлива: факт vs ГСМ-норма
•	Штрафы: сумма, по ТС/водителям
•	Рейтинг водителей
2.5. Дашборд механика
•	Очередь ТС на осмотр
•	Карточка ТС: история, неисправности, пробег, ТО
•	Чек-лист: тормоза, шины, свет, жидкости, кузов, огнетушитель, аптечка, тахограф
•	ОК / Неисправность (комментарий + фото)
•	Допущен (ЭП) / Не допущен (причина → автозаявка на ремонт)
•	Светофор сроков: техосмотр, ОСАГО, ТО, тахограф, пропуска
•	Журнал осмотров: неизменяемый, 3+ лет
2.6. Дашборд медика
•	Очередь водителей
•	Карта здоровья, предыдущие осмотры
•	Чек-лист: АД, пульс, температура, состояние, опьянение, жалобы
•	Допущен (ЭП) / Не допущен → уведомление диспетчеру
•	Контроль медсправок: уведомление за 30 дней
•	Журнал: неизменяемый, доступ медик + руководитель
•	Статистика недопусков
2.7. Дашборд ремонтной службы
•	Канбан: Создана → Ждёт з/ч → В работе → Готово
•	Описание, фото, приоритет, исполнитель
•	Склад запчастей (упрощённый)
•	Фиксация работ: вид, запчасти, стоимость, пробег
•	История по ТС, плановое ТО
•	Итоги затрат по ТС и периодам
2.8. Дашборд бухгалтера
•	Реестр рейсов для выставления счёта
•	Автоматический расчёт стоимости по договору (за км/тонну/час/маршрут)
•	Формирование счетов и актов/УПД
•	Реестр штрафов: привязка к ТС/водителю, статус оплаты
•	Сверка с клиентами
•	Экспорт для 1С
2.9. Вайрфреймы ключевых экранов
Для ускорения разработки рекомендуется подготовить вайрфреймы (low-fidelity мокапы) следующих 7 экранов до начала вёрстки. Вайрфреймы оформляются как отдельный артефакт (Figma / React-прототипы).
•	Канбан-доска логиста (поток заявок по статусам)
•	Карта диспетчера (ТС на карте + панель назначения + таймлайн)
•	Чек-лист механика на планшете (осмотр ТС с фото и подписью)
•	Карта медосмотра на планшете (показатели + подпись)
•	Канбан ремонтной службы (заявки по статусам)
•	KPI-панель руководителя (графики, метрики, светофоры)
•	Экран водителя — мобильное приложение (задание, маршрут, подтверждение доставки)
 
3. Бизнес-процессы
3.0. Приоритеты требований (MoSCoW)
Каждый бизнес-процесс и функциональный блок имеет приоритет, определяющий очерёдность реализации.
Приоритет	Что входит	Фаза
MUST (без этого не запуск)	Приём заявок и назначение рейсов; Предрейсовый техосмотр ТС (журнал + чек-лист + ЭП); Предрейсовый медосмотр (журнал + чек-лист + ЭП); Формирование путевого листа; Исполнение рейса (карта, контроль); Подтверждение доставки; Мобильное приложение водителя с оффлайном; Дашборды: логист, диспетчер, механик, медик	1

SHOULD (важно для полного контура)	Дашборд руководителя (KPI); Дашборд бухгалтера (тарификация, счета); Контроль РТО по тахографу; Учёт штрафов ГИБДД; ГСМ-нормы; Возвраты и претензии; Коннектор 1С	3

WON'T (не входит)	Бухучёт, кадры, зарплата; Складские операции; Прямое управление бортовыми системами	—
3.1. Приём заявки
Дашборд: логист.
1.	Клиент создаёт заявку (или логист по звонку/email/из шаблона): груз (вес, объём, места), адреса, окна, требования к ТС.
2.	Система проверяет поля, присваивает номер, генерирует план документов.
3.	Проверка веса vs грузоподъёмность подходящих ТС.
4.	Логист подтверждает или уточняет.
5.	Передаётся диспетчеру.
•	Шаблоны для постоянных клиентов — создание в 2 клика.
•	Сквозной уникальный идентификатор.
3.2. Планирование и назначение рейса
Дашборд: диспетчер.
1.	Диспетчер формирует рейс: маршрут, точки, окна.
2.	Назначает ТС и водителя (только доступные).
3.	Автопроверки: грузоподъёмность, пропуска для маршрута, остаток РТО, совместимость заявок.
4.	Несколько заявок → один рейс.
5.	Статус «Назначен».
•	Нельзя: просроченные документы, без допуска, без меддопуска, недостаток РТО.
•	Нет пропуска — предупреждение (мягкая блокировка). Перевес — жёсткая блокировка.
3.3. Предрейсовый техосмотр ТС
Дашборд: механик.
1.	Очередь ТС на осмотр.
2.	Чек-лист (настраиваемый): тормоза, шины, свет, жидкости, кузов, огнетушитель, аптечка, тахограф.
3.	ОК / Неисправность (комментарий + фото).
4.	Проверка сроков документов (техосмотр, ОСАГО, тахограф, пропуска).
5.	Допущен (ЭП) / Не допущен (причина + автозаявка на ремонт).
6.	Запись в журнал (неизменяемый, 3+ лет). Данные → путевой лист.
3.4. Предрейсовый медосмотр
Дашборд: медик.
1.	Очередь водителей.
2.	Чек-лист: АД, пульс, температура, состояние, опьянение, жалобы.
3.	Допущен (ЭП) / Не допущен (комментарий → диспетчеру).
4.	Журнал (неизменяемый, 3+ лет, персональные данные). Данные → путевой лист.
5.	Контроль медсправок: автоуведомление за 30 дней.
3.5. Формирование путевого листа
Дашборд: диспетчер.
1.	После допусков (механик + медик) система формирует путевой лист автоматически.
2.	Содержит: организация, ТС (госномер, марка, одометр), водитель (ФИО, ВУ), штамп механика (допуск, подпись, время), штамп медика, маршрут, задание (номера заявок).
3.	Номер привязан к рейсу.
4.	По возвращении: одометр, расход топлива, время возврата.
5.	Подписывается ЭП, хранится как ЭПД.
•	Не формируется без обоих допусков.
•	Хранение 5+ лет. С сентября 2026 — формат ЭПД обязателен.
3.6. Погрузка и выпуск на линию
Дашборд: диспетчер + водитель.
1.	Проверка готовности: допуски + путевой лист.
2.	Водитель: сверка груза, фото, подписание ТТН/ЭТрН. Расхождения — акт.
3.	Рейс → «В пути». Клиенту: «Груз загружен».
3.7. Исполнение рейса
Дашборд: диспетчер.
1.	Телематика: геопозиция, скорость, топливо.
2.	Тахограф: время вождения/отдыха в реальном времени.
3.	Отклонения → уведомление.
4.	Лимит РТО → предупреждение водителю и диспетчеру.
5.	Точки выгрузки: прибытие, передача, подпись, фото.
6.	Клиенту: «В пути, ETA ...», «Доставлен».
3.8. Закрытие рейса
Дашборд: логист + диспетчер.
1.	Водитель завершает. Проверка комплектности.
2.	Одометр, остаток топлива → закрытие путевого листа.
3.	Рейс → «Завершён». Заявки → «Доставлена»/«Возвращена».
4.	Послерейсовый осмотр (опционально).
5.	Рейс доступен для тарификации.
3.9. Тарификация и счета
Дашборд: бухгалтер.
Базовые типы тарифов
•	За километр (ставка × расстояние по маршруту)
•	За тонну (ставка × вес груза)
•	За час (ставка × время от погрузки до выгрузки)
•	Фиксированная ставка за маршрут
•	Комбинированный (фикс + за км сверх порога)
Модификаторы и доплаты
•	Простой на погрузке/выгрузке: бесплатный лимит (например, 2 часа), далее — почасовая ставка.
•	Дополнительная точка: фиксированная доплата за каждую доп. точку маршрута.
•	Ночная доставка: коэффициент (например, ×1.5) за доставку в период 22:00–06:00.
•	Срочная доставка: коэффициент за заявку с окном менее 4 часов.
•	Возврат груза: оплата обратного рейса (настраиваемый % от прямого).
•	Отмена заявки: штраф клиенту при отмене менее чем за N часов (настраивается по договору).
•	Выходные/праздники: коэффициент.
Правила расчёта
•	НДС: 20% (общая система) или без НДС (УСН). Настраивается по организации.
•	Минимальная стоимость рейса: настраивается по договору.
•	Округление: до 1 ₽ / до 10 ₽ / до 100 ₽ (настраивается).
•	Валюта: рубль (мультивалютность — фаза 4).
•	Мультидоговор: у одного клиента может быть несколько договоров с разными тарифами.
Формируемые документы
•	Счёт на оплату (по закрытым рейсам за период).
•	Акт выполненных работ (детализация по рейсам).
•	УПД (для ЭДО).
•	Реестр рейсов (приложение к акту).
3.10. Ремонты и ТО
Дашборд: ремонтная служба.
1.	Заявка: автоматически (недопуск), водителем, механиком, по графику.
2.	Создана → Ждёт з/ч → В работе → Готово.
3.	Фиксация: работы, запчасти, стоимость, пробег.
4.	ТС → «Доступно».
3.11. Автопарк и водители
•	Учёт ТС с историей
•	Сроки: техосмотр, ОСАГО, ТО, тахограф, пропуска
•	Расход топлива: факт vs ГСМ-норма (сезон, загрузка, тип ТС)
•	Пробег по телематике
•	Автоуведомления, блокировка при просрочке
•	Водители: допуски, РТО, медсправки, привязка к ТС, штрафы
3.12. Требования к мастер-данным и справочникам
Цель: обеспечить качество данных с первого дня, избежать «Excel-ада».
•	Дубликаты: система предупреждает при создании контрагента с существующим ИНН или ТС с существующим госномером.
•	Импорт: массовая загрузка из Excel/CSV при первоначальном наполнении.
•	Архивирование: удалённые записи не удаляются физически, а переходят в статус «Архив».

3.18. Оффлайн-режим и разрешение конфликтов
Цель: обеспечить работу водителя без интернета до 72 часов.
Принцип работы
Мобильное устройство ведёт локальный журнал событий. Каждое событие получает локальный UUID и timestamp устройства. При появлении связи — все события передаются на сервер без потерь (append-only).
Модель разрешения конфликтов
•	События водителя всегда сохраняются (append-only, не перезаписываются).
•	Конфликт возникает на уровне агрегатов: например, диспетчер переназначил рейс, пока водитель оффлайн уже отметил погрузку.
•	Оба варианта сохраняются в журнале. НЕ используется last-write-wins.
•	Конфликтные пары попадают в очередь ручного разбора диспетчера.
•	Диспетчер видит оба варианта с таймлайном и принимает решение.
•	Максимальное время оффлайн: 72 часа (настраивается). После 72 часов — принудительная блокировка до синхронизации.
Типовые сценарии конфликтов
•	Переназначение рейса во время оффлайна водителя → водитель продолжал старый рейс → диспетчер разбирает.
•	Отмена заявки клиентом, пока водитель уже в пути без связи → при синхронизации диспетчер решает (возврат или доставка).
•	Два водителя оффлайн отметили одну точку → дубль фиксируется, диспетчер удаляет лишнее.
•	Механик изменил статус ТС (неисправно), пока водитель оффлайн выехал → при синхронизации: инцидент, уведомление руководителю.

R = чтение, W = создание/редактирование, — = нет доступа. Для медданных действуют дополнительные ограничения (см. А.2).

А.2. Режим медицинских данных (152-ФЗ)
Медицинские данные (показатели АД, пульс, температура, жалобы, история осмотров) — персональные данные специальной категории.
•	Полный доступ: только медик (RW на свои записи) и руководитель (R, только статистика).
•	Диспетчер: видит ТОЛЬКО факт «Допущен / Не допущен» + время. НЕ видит показатели, причины недопуска, историю.
•	Руководитель: видит агрегированную статистику (кол-во допусков/недопусков за период, причины — категории без персональных деталей). Доступ к конкретным записям — по запросу с логированием.
•	Все остальные роли: нет доступа к медданным.
•	Аудит доступа: каждое обращение к медданным логируется (кто, когда, к чьим данным, какое действие). Лог неизменяемый, хранение 3+ лет.
•	Согласие на обработку: при регистрации водителя система фиксирует дату подписания согласия. Без согласия — медосмотр в системе невозможен.
•	Основание обработки: ст. 213 ТК РФ (обязательные медосмотры), Приказ Минздрава №835н.
А.3. Администратор
•	Управляет ролями, пользователями, справочниками, шаблонами чек-листов.
•	НЕ имеет доступа к медданным, финансовым показателям, KPI.
•	Может назначать/отзывать роли, но не может сам себе добавить роль «Руководитель» или «Медик» (защита от эскалации привилегий).
 
Приложение Б. Каталог событий (Event Types)
Б.1. Принципы
•	Каждое событие — неизменяемая запись в append-only журнале.
•	Поля: id (UUID v7), timestamp (серверное), author_id, author_role, event_type, entity_type, entity_id, data (JSONB), version.
•	Идемпотентность: повторная отправка события с тем же id игнорируется (для оффлайн-синхронизации).
•	Версионирование: при конфликте (оффлайн) оба события сохраняются с пометкой conflict=true.

Приложение В. Критерии приёмки MVP (фаза 1)
В.1. Заявки и рейсы
•	GIVEN логист авторизован WHEN создаёт заявку с обязательными полями THEN заявка получает статус «Черновик», уникальный номер, план документов.
•	GIVEN заявка «Подтверждена» WHEN диспетчер назначает ТС и водителя THEN система проверяет: ТС исправно + документы в порядке + водитель с допуском + грузоподъёмность >= вес груза. При нарушении — блокировка с указанием причины.
•	GIVEN ТС с просроченным ОСАГО WHEN диспетчер пытается назначить его THEN система блокирует назначение и показывает причину.
•	GIVEN водитель без меддопуска на сегодня WHEN диспетчер пытается назначить его THEN система блокирует.
•	GIVEN несколько заявок на один маршрут WHEN диспетчер объединяет их THEN создаётся один рейс с несколькими точками, суммарный вес проверяется.
В.2. Предрейсовый техосмотр
•	GIVEN ТС назначено на рейс WHEN механик открывает дашборд THEN ТС появляется в очереди на осмотр.
•	GIVEN механик заполнил чек-лист, все пункты ОК WHEN нажимает «Допустить» THEN создаётся акт с ЭП механика, событие inspection.tech_completed, ТС получает допуск.
•	GIVEN в чек-листе есть пункт «Неисправность» WHEN механик нажимает «Не допустить» THEN создаётся акт + автоматическая заявка на ремонт + уведомление диспетчеру.
•	GIVEN акт осмотра создан WHEN кто-либо пытается изменить его THEN система отклоняет (append-only).
•	GIVEN техосмотр ТС не пройден WHEN диспетчер пытается выпустить рейс THEN система блокирует с сообщением «Нет допуска механика».
В.3. Предрейсовый медосмотр
•	GIVEN водитель назначен на рейс WHEN медик открывает дашборд THEN водитель в очереди.
•	GIVEN медик заполнил показатели, всё в норме WHEN нажимает «Допустить» THEN акт с ЭП, событие, допуск.
•	GIVEN медик не допустил THEN уведомление диспетчеру (без деталей показателей), диспетчер видит только «Не допущен».
•	GIVEN диспетчер авторизован WHEN запрашивает медданные водителя (АД, пульс) THEN система отказывает в доступе.
•	GIVEN водитель без согласия на обработку перс. данных WHEN медик пытается создать осмотр THEN система блокирует с сообщением.
В.4. Путевой лист
•	GIVEN ТС допущено (механик) И водитель допущен (медик) WHEN система формирует путевой лист THEN документ содержит все обязательные реквизиты, штампы, номер.
•	GIVEN нет допуска механика ИЛИ нет допуска медика WHEN система пытается сформировать путевой лист THEN формирование невозможно.
•	GIVEN путевой лист сформирован THEN его номер уникален и принадлежит непрерывной нумерации.
В.5. Исполнение рейса
•	GIVEN рейс в статусе «В пути» WHEN GPS передаёт координаты THEN точка появляется на карте диспетчера в течение 30 секунд.
•	GIVEN водитель прибыл на точку выгрузки WHEN подтверждает доставку (фото + подпись) THEN событие checkpoint_completed, заявка → «Доставлена», клиенту — уведомление.
•	GIVEN водитель оффлайн 4 часа WHEN появляется связь THEN все события синхронизируются, порядок сохраняется по локальным timestamps.
•	GIVEN рейс завершён, все точки пройдены WHEN водитель нажимает «Завершить» THEN рейс → «Завершён», путевой лист закрывается (одометр, время).
В.6. Мобильное приложение
•	GIVEN водитель авторизован WHEN открывает приложение THEN видит список назначенных рейсов на сегодня.
•	GIVEN нет интернета WHEN водитель подтверждает доставку THEN данные сохраняются локально, синхронизируются при связи.
•	GIVEN оффлайн > 72 часов THEN приложение показывает предупреждение и блокирует новые действия до синхронизации.
•	GIVEN водитель WHEN пытается открыть чужой рейс THEN система отказывает в доступе.
В.7. Дашборды
•	GIVEN логист авторизован THEN видит канбан заявок с актуальными статусами, время отклика < 2с.
•	GIVEN диспетчер авторизован THEN видит карту с ТС, обновление позиций < 30с.
•	GIVEN механик авторизован THEN видит очередь ТС с историей и сроками.
•	GIVEN медик авторизован THEN видит очередь водителей. НЕ видит финансы, ремонты, заявки.
•	GIVEN пользователь с ролью «Водитель» WHEN заходит в веб-панель THEN доступны только его рейсы и документы.
В.8. Неизменяемый журнал
•	GIVEN любое действие в системе THEN в журнале появляется событие с UUID, timestamp, автором, данными.
•	GIVEN событие записано WHEN кто-либо пытается UPDATE или DELETE THEN БД отклоняет операцию.
•	GIVEN 100 000 событий в журнале WHEN запрос на чтение последних 1000 THEN ответ < 500мс.
