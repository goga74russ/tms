## Fleet и Repairs — глубокая проверка бизнес-инвариантов

### Fleet: текущая «истина» по доступности ТС и водителя
В текущем коде «истина» о статусе ТС в основном живёт в `vehicles.status`, и модуль Repairs напрямую меняет этот статус при создании/закрытии ремонта. citeturn46view1turn44view0  
Также Fleet обогащает ТС рассчитанными «дедлайнами» документов и флагом блокировки (для UI/решений), но это выглядит как application-level логика без жёстких DB-инвариантов. citeturn44view0

**Проблема HIGH: отсутствие объектного/рольного RLS на Fleet для driver**  
RBAC даёт `driver` права `read Driver` и `read Vehicle` (комментарий: «только свои данные — фильтрация на уровне query»). citeturn74view0  
Но `GET /fleet/drivers` и `GET /fleet/drivers/:id` никак не фильтруют данные «только свой водитель», а просто возвращают список/объект. citeturn70view0  

Impact: утечка персональных данных водителей (ФИО, даты, номера ВУ, согласие на ПДн и т.п. — зависит от схемы) любому водителю.  
Root cause: RBAC предполагает query-level RLS, но route/service его не реализуют. citeturn74view0turn70view0  
Fix:  
* Вариант A (быстрее и безопаснее): убрать у `driver` глобальные `read Driver` и заменить на отдельный endpoint `GET /me/driver` (или `GET /fleet/drivers/me`) с принудительным `where drivers.userId = request.user.userId`.  
* Вариант B: оставить `read Driver`, но во всех `fleet/drivers*` ветках добавлять RLS: если роль driver — разрешать только `myDriverId`.  
How to test: интеграционный тест Fastify inject: логин driver1, запрос `/fleet/drivers`, ожидаем либо 403, либо список из 1 элемента (только self). citeturn70view0turn59view1

### Repairs: статусная модель и её влияние на доступность ТС
В Repairs есть явная state machine:
`created → waiting_parts → in_progress → done` и валидатор переходов в `updateRepairStatus()`. citeturn46view1  
При создании ремонта сервис переводит `vehicles.status` в `maintenance` или `broken` (в зависимости от priority) транзакционно вместе с созданием repair request и записью событий. citeturn46view1  
При переводе ремонта в `done` сервис возвращает ТС в `available`, тоже транзакционно. citeturn46view1

**Проблема HIGH: некорректная «разблокировка» ТС при наличии других активных ремонтов**  
Сейчас при `repair -> done` ТС ставится `available` без проверки, есть ли другие ремонты в статусах `created/waiting_parts/in_progress`. citeturn46view1  

Impact: ТС может стать «доступным» при реально незакрытых работах → поломка инварианта “vehicle availability”.  
Root cause: vehicle.status моделируется как производная от последнего события ремонта, но нет агрегации по всем активным repair_requests. citeturn46view1  
Fix: при `done` пересчитывать vehicle.status: если есть любой активный ремонт — оставлять `maintenance/broken` (в зависимости от max severity), иначе `available`. Лучше: сделать status производным (projection) из событий ремонта, а не ручным полем.  
How to test: создать два ремонта на одно ТС, закрыть один → ожидать, что статус остаётся maintenance/broken. citeturn46view1

**Проблема MED: множественные ремонты на одно ТС без жёсткого ограничения**  
Для «scheduled» ремонтов есть защита от дублей (по source + активным статусам) только в `checkScheduledMaintenance()`, но для ручных/авто ремонтов ограничений нет. citeturn46view1turn19view0  

Impact: гонки/дубли заявок, сложности в учёте и доступности.  
Fix: если бизнес-правило “один активный ремонт на ТС” — добавить уникальный partial index (vehicle_id WHERE status IN (...)) и обработку 409 на вставке. Если правило другое — явно зафиксировать policy (например допускаются параллельные «мелкие» заявки).  
How to test: конкурентно создать 2 repairs на одно ТС → одна должна упасть 409.

## Очереди, синхронизация и append-only журнал

### BullMQ: конфигурация очередей и повторяемых задач
Очереди `wialon-sync` и `fines-sync` создаются с `attempts=3` и экспоненциальным backoff, плюс чистка повторяемых задач на рестарте. citeturn50view0turn37view0  
Worker’ы стартуют только если Redis доступен, иначе явно отключаются. citeturn37view0turn51view0

**Проблема HIGH: отсутствие outbox/транзакционного enqueue**
Сейчас постановка задач в очередь не связана транзакционно с изменениями в Postgres (enqueue происходит напрямую через BullMQ). citeturn50view0turn55view0  
Impact: при сбоях между commit и enqueue (или наоборот) возможны «потерянные» синки/двойные выполнения.  
Fix: outbox pattern: запись команды в Postgres + отдельный publisher, который публикует job с `jobId=commandId` (dedupe).  
How to test: integration test: эмулировать падение после commit и до enqueue → outbox publisher должен догнать.

### Worker’ы и аудит событий: системный пользователь и целостность
В воркерах используется фиктивный `SYSTEM_USER_ID = 0000...` и затем пишется `recordEvent(...)`. citeturn53view0turn53view1  
Но в seed пользователи создаются без явного задания id (генерируются автоматически), и системного пользователя с этим id seed не создаёт. citeturn54view0  
При этом миграция накладывает FK `events.author_id -> users.id`. citeturn36view2  

**Проблема HIGH: воркеры могут падать на записи event из-за FK**
Impact: «косвенная» деградация — side effects уже применены (например fine insert), а job падает на audit event → ретраи, шум, непредсказуемые состояния интеграций. citeturn53view1turn22view0turn36view2  
Root cause: system actor не представлен в users при наличии FK.  
Fix:  
1) Создать системного пользователя в миграции/seed с фиксированным id (0000...) и ролью `system`, либо  
2) Разрешить `author_id` nullable/без FK (хуже для аудита), либо  
3) Писать события от реального сервисного пользователя (например `admin`/`integration_bot`) и гарантировать его наличие.  
How to test: поднять чистую БД, миграции, затем запустить worker вручную (или `triggerFinesSync`) — он должен завершаться SUCCESS без FK ошибок. citeturn55view0turn71view3

### Append-only журнал (events)
Триггер `prevent_event_modification()` запрещает UPDATE/DELETE, разрешая только изменение `conflict` при полном совпадении остальных полей. citeturn38view0turn22view0  
Триггеры применяются при старте сервера через `rawSql.unsafe(APPEND_ONLY_TRIGGER_SQL)`. citeturn37view0turn38view0  

**Проблема HIGH: idempotency externalId не доведён до БД**
* Код “умеет” externalId. citeturn22view0  
* База в миграции не имеет externalId. citeturn34view2turn35view0  

Impact: либо фича не работает, либо при попытке использовать — падение вставки.  
Fix: добавить `external_id` (varchar/uuid), уникальный индекс, и обновить триггер, чтобы external_id тоже сравнивался при разрешённом update conflict.  
How to test: дважды вызвать `recordEvent({externalId: 'X'})` → второй раз должен вернуть `null` без ошибок.

### Offline sync и идемпотентность команд
`processSyncEvents()` проверяет, что пользователь driver и что он — назначенный водитель рейса (verifyTripOwnership). Это сильная защита от IDOR в sync. citeturn59view0turn59view1

**Проблема HIGH: повторная доставка sync-команд может дублировать side effects**
Impact: повторная смена статуса/повторное «arrived/completed» может либо ломать 상태-машину, либо тихо перезаписывать timestamps/артефакты (зависит от `updateRoutePoint`), а также плодить события. citeturn59view0turn22view0

### Finance: корректность биллинга по рейсам
`generateInvoices()` выбирает `trips` через `innerJoin(orders, eq(trips.id, orders.tripId))` и вытаскивает только `trips.id`. citeturn58view0

### Frontend: клиент API и хранение токена
