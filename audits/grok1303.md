1. Executive Summary
Проект TMS реализует 80–85% заявленной бизнес-логики по TZ_TMS.MD (версия 3.0, март 2026). Ядро сквозного процесса (заявка → рейс → осмотры → путевой лист → исполнение → закрытие → биллинг) реально работает и защищено жёсткими проверками в сервисах.
Главный сильный аспект:

Строгая стейт-машина TRIP_TRANSITIONS + canTransition в apps/api/src/modules/trips/service.ts.
Обязательный блок путевого листа без сегодняшних допусков механика и медика в apps/api/src/modules/waybills/service.ts:generateWaybill (проверка hasValidTechInspectionToday + hasValidMedInspectionToday, throw 409).
Автоматическое обновление статусов связанных сущностей в одной транзакции + event-journal.

Главный риск:
Отсутствие контроля РТО по тахографу (Приказ №424) и полного процесса претензий/возврата оригиналов документов. Система пока не готова к реальной эксплуатации с собственным автопарком без доработок — можно вести 1–2 рейса в день в тестовом режиме, но при 10+ ТС начнутся нарушения законодательства и операционные сбои.
Уровень зрелости бизнес-логики: MVP+ (фаза 1–2 TZ завершены, фаза 3 — частично).
2. Что реально реализовано по бизнес-логике

Полный сквозной флоу от заявки до биллинга с валидацией переходов.
Жёсткие блоки в assignTrip (перевес, просрочки документов ТС/водителя, blocking-incidents).
Путевой лист не формируется без двух осмотров (Zod + сервис).
Event-journal (append-only + recordEvent в каждой мутации).
RBAC + CASL (10 ролей), RLS для водителя/механика.
Тарификация (7 модификаторов) и генерация счёта/акта/УПД в modules/finance.
Оффлайн-sync (WatermelonDB) + конфликты в очереди.

3. Что заявлено в TZ_TMS.MD, но не реализовано / частично

Контроль РТО (остаток часов, макс 9 ч/сутки) — только soft-warning в assignTrip, нет реального парсинга тахографа (только мок Wialon).
Полный процесс претензий/расхождений/возврата оригиналов — нет сущности Claim, нет статуса документов «возвращён оригинал».
ЭПД в продакшен-формате (только прототип etrn-generator.ts).
Интеграции (ГИБДД штрафы, Платон, реальный тахограф) — все на моках.
Автоуведомления клиенту по всем статусам — только Telegram-бот (не настроен).

5. Сквозной аудит процесса “от заявки до закрытия рейса”
Приём заявки
Сущности: Order (Zod), статусы из enums.
Реализовано: CRUD + confirm в modules/orders. Переход draft → confirmed.
Планирование рейса + назначение
Сущности: Trip, Vehicle, Driver.
Проверки: вес, доступность, документы, incidents (hard).
Реализовано: assignTrip полностью.
Предрейсовые осмотры
Сущности: TechInspection, MedInspection (с чек-листами, подписями).
Реализовано: очереди + decision: approved/rejected.
Выпуск (путевой лист)
Критически важно: generateWaybill блокирует создание без двух сегодняшних допусков + incidents. Связь: techInspectionId, medInspectionId обязательны в схеме и сервисе.
Движение рейса
Статусы + changeTripStatus с переходами. Обновление заказов.
Контроль доставки / инциденты
routePoints + incidents (blocksRelease).
Закрытие + биллинг
COMPLETED → BILLED. Авто-расчёт в finance.
Документы
Waybill + etrn-generator. Нет контроля возврата оригиналов.
6. Нарушения бизнес-инвариантов (найдены в коде)

Можно создать рейс без заявки (tripId без orders).
Медсправка и тахограф — только soft-warning (можно обойти).
Нет проверки остатка РТО при назначении.
Admin-роль обходит все RLS (CASL can('manage','all')).
Event-journal позволяет частичные UPDATE (gpt0503).
Можно закрыть рейс без подтверждения доставки (нет mandatory routePoints status).

7. Проблемы модели данных
Модель адекватна (Drizzle + FK: trip → orders, waybill → inspections). Нет смешения заявка/рейс. Но:

Нет таблицы Claim / DocumentReturn.
Нет RtoLog для тахографа.
numeric(12,2) для денег — ок, но real для пробега (было исправлено в аудите).

8. Проблемы статусов и переходов
Статус-машина есть и работает (TRIP_TRANSITIONS). Но нет FSM-библиотеки — только map + функция. Переходы валидируются в сервисе.
9. Проблемы ролей и разграничения
10 ролей + CASL. Водитель видит только свои рейсы. Но механик/медик могут обойти диспетчера (нет блокировки выпуска без путевого листа на UI). Admin — суперпользователь.
10. Проблемы документов и закрытия
Путевой лист — ок. Но нет контроля возврата ТТН/ЭТрН оригиналов и связи с актом. Рейс можно закрыть без всех документов.
11. Топ-10 критических бизнес-рисков

Нарушение Приказа №424 (РТО) — штрафы ГИБДД.
Выпуск ТС без медосмотра (мягкий блок).
Неполный append-only журнал → потеря доказательной базы.
Отсутствие претензий → споры с клиентами.
Моки интеграций → реальные рейсы сломаются.
Admin bypass 152-ФЗ.
Нет возврата документов → проблемы с 1С/бухгалтерией.
Оффлайн-синхронизация не протестирована >24 ч.
UUID не v7 в событиях.
KPI с fallback-именами (Sprint 4.1).

13. Вывод: насколько проект годен как реальная TMS
Не годен в текущем виде для коммерческой эксплуатации (можно запустить пилот на 2–3 ТС).
Блокирует: отсутствие РТО-контроля, претензий, возврата документов и реальных интеграций.
При доработках 1–5 из топа — можно запускать через 3–4 недели.
