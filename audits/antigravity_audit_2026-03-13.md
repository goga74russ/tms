# КОМПЛЕКСНЫЙ АУДИТ БИЗНЕС-ЛОГИКИ TMS
**Дата:** 13 марта 2026
**Аудитор:** Antigravity AI
**Статус проекта:** Post-MVP / Pre-Production

## 1. Executive Summary

Проект обладает на удивление крепким ядром: архитектурная связка "Предрейсовые осмотры → Блокировки → Путевой лист" реализована на уровне выше среднего по рынку. Используются правильные паттерны (идемпотентность, транзакции, append-only журнал событий с защитой на уровне SQL).

Однако система страдает от "болезни роста" — отсутствуют важные бизнес-инварианты на стыке модулей. Основные проблемы кроются в тарификации мультизаявочных рейсов, статусной модели рейсов и отсутствии контура претензионной работы и возврата документов. Без их решения запуск реальных коммерческих рейсов приведет к финансовым потерям и юридическим рискам.

## 2. Топ-6 Критических проблем (P0 - Блокеры запуска)

### 2.1. Ошибки в тарификации (Финансовые потери)
* **Проблема:** В `calculateTripCost` (`tarification.service.ts`) используются несуществующие поля схемы: `plannedCompletionAt` (вместо `plannedArrivalAt`), `returnRequired` отсутствующее в схеме заявок, и `nightMultiplier` (вместо `nightCoefficient`).
* **Результат:** `urgentCost` и `returnCost` всегда равны 0. Ночные надбавки игнорируются. Инвойсы (счета клиентам) систематически занижаются.
* **Мульти-заявки:** Тарификация берет только первый заказ через `trips.with({ order })`. Для сборных грузов (один рейс — несколько заказов) счет будет категорически неверным.

### 2.2. Заказы не переводятся в 'delivered'
* **Проблема:** В `trips/service.ts:changeTripStatus(COMPLETED)` обновляются статусы ТС и путевых листов, но связанные заказы (`orders`) остаются в статусе `in_transit`.
* **Результат:** Сломанный жизненный цикл заказа, заказы зависают в системе, недовольство клиентов (если бы клиентский портал их отображал).

### 2.3. Обход выпуска через прямую смену статуса
* **Проблема:** Универсальная ручка `/trips/:id/status` позволяет перевести рейс напрямую в `waybill_issued` без проверки `waybillId` (в обход `generateWaybill`).
* **Результат:** Диспетчер может "выпустить" машину на линию без прохождения механика и медика, ломая главный Legal/Compliance барьер проекта.

### 2.4. 'Отмена' не освобождает транспорт
* **Проблема:** При переводе рейса в статус `cancelled`, назначенное транспортное средство (vehicle) не получает статус `available`, оставаясь заблокированным.
* **Результат:** Искусственный дефицит автопарка — машины "зависают" в отмененных рейсах.

### 2.5. Несогласованность 'закрытия'
* **Проблема:** Закрытие путевого листа (`closeWaybill`) обновляет одометр, но не переводит рейс в статус `completed`. При этом финансовый биллинг собирает в счета только `completed` рейсы.
* **Результат:** Зависание рейсов между операционным и финансовым контурами. Счета клиентам могут не выставиться.

### 2.6. Риски безопасности (XSS и RBAC)
* **Проблема:** Хранение JWT-токенов в `localStorage` на фронтенде делает систему уязвимой для XSS. К тому же, у роли `client` в CASL указано право `manage` (`create/read/update/delete`) на `Order`, хотя должен быть только `read`.

## 3. Серьезные архитектурные и бизнес-разрывы (P1)

1. **Нет контроля режима труда и отдыха (РТО):** `assignTrip` проверяет страховки, но не проверяет часы вождения. Допуск усталых водителей — прямой путь к многотысячным штрафам ГИБДД (Приказ Минтранса №424).
2. **Слабый контур работы с документами:** Нет доменных сущностей для отслеживания возврата оригиналов ТТН/ЭТрН. Невозможно выстроить жесткое правило "нет оригиналов = рейс не закрыт в финансах".
3. **Отсутствие модуля Претензий (Claims):** Споры, недостачи или порча груза не имеют финансового и процессуального выражения в системе (Workflow расследования отсутствует).
4. **Прицепы (Trailers):** Сущность добавлена в схему в Sprint 9, но не привязана к сущности `trips` (`FK trailerId`). Невозможно назначить тягач с прицепом как единую сцепку.
5. **GPS и Интеграции — это заглушки:** Использование `WialonMock` и `DaDataMock`. Расчет расстояний по координатам детерминирован и не отражает реальность, что делает невозможным тарификацию "по километрам".
6. **Ошибки в PDF-формах (ТТН):** В форме 1-Т (`orders/routes.ts`) для грузополучателя передается адрес вместо "Наименования юр. лица". Такой документ юридически ничтожен.

## 4. Action Plan: План приоритизированных исправлений

### Sprint 1: Rapid Fixes (Срочные багфиксы)
*Высокий ROI: занимает 1-2 дня, устраняет критические бизнес-дыры, спасая "поток ценности".*
* [ ] **Тарификация (`finance`):** Исправить имена полей (`plannedArrivalAt`, `nightCoefficient`) и переписать логику расчета с учетом массива `trip_orders`, чтобы сборные грузы считались корректно.
* [ ] **Смена статусов (`trips`):** В метод `changeTripStatus(COMPLETED)` добавить `UPDATE orders SET status = 'delivered'`.
* [ ] **Освобождение активов (`trips`):** В `changeTripStatus(CANCELLED)` добавить логику возврата ТС / Прицепа в `available`.
* [ ] **Закрытие лазейки (`trips`):** В `changeTripStatus` добавить Guard на переход в `waybill_issued` (запрет, если путевой лист физически не был создан).
* [ ] **Реквизиты в ТТН (`orders`):** В генераторе PDF ТТН подтягивать `consigneeName` из данных Контрагента, а не использовать строку адреса.
* [ ] **RBAC (`auth`):** Для роли `client` в `rbac.ts` понизить права с `manage` до `read` для сущности `Order`.

### Sprint 2: Core Enhancements (Отвердение ядра)
*Занимает 1 неделю. Делает систему пригодной для надежного внутреннего тестирования автопарком.*
* [ ] **Безопасность:** Перевести авторизацию (JWT) с `localStorage` на `httpOnly` cookies.
* [ ] **Сцепка с прицепами:** Добавить в БД `trailerId` для `trips`, добавить проверки прицепов в функцию `assignTrip()`, выводить прицеп в Путевом Листе.
* [ ] **Сквозное закрытие:** Добавить метод `openWaybill()` (для диспетчера) и синхронизировать `closeWaybill` с валидным завершением рейса (запрет на закрытие, если точки не пройдены).
* [ ] **Блокировки ТС и Водителей:** Добавить жесткий блок (Hard Block) на проверку окончания медсправки (`medCertExpiry`) в логику подбора водителя. Внедрить авто-статус `maintenance` для ТС при заваленном техосмотре.

### Sprint 3: Business Completeness (Бизнес-полнота для Продакшена)
*Занимает 2-4 недели. Превращает MVP в полноценную TMS Enterprise-уровня.*
* [ ] **Интеграция RTO (Тахограф):** Полный парсинг данных телематики, модуль расчета остатка часов и блокировки назначений при переработках.
* [ ] **Реестр документов (Paper Trails):** Модуль контроля возврата оригиналов подписанных ТТН/УПД с блокировкой финансового закрытия рейса без полного комплекта.
* [ ] **Модуль претензий и инцидентов (Claims):** Workflow расследований (создание, расследование, привязка к рейсу, расчет финансовых удержаний и корректировок).
* [ ] **Реальные интеграции:** Отказ от мок-сервисов (Wialon, ГИБДД).
* [ ] **ЭТрН / ЭПД:** Доработка текущего прототипа генератора XML до промышленного стандарта (интеграция с провайдером ЭДО, реализация подписи УКЭП).
