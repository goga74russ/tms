### Как оно работает сейчас

Ниже — “как устроено сейчас” (и рядом — риски/вопросы, которые сразу видны по коду):

1) Монорепа управляется pnpm; в корне есть скрипты `dev`, `build`, `lint`, `test`. citeturn3view0turn4view0  
2) `dev` поднимает API и WEB параллельно через `concurrently`. citeturn3view0

## Собираемость и CI

### Соберётся ли сейчас?

Собираемость “в целом” выглядит близкой к рабочей (Dockerfile’ы есть, `build`-скрипты есть, Next настроен на standalone). citeturn21view0turn48view0turn49view1turn3view0

Но “гарантированно собирался (CI + локально + Docker)” — сейчас нет, по двум причинам:

1) Root `pnpm test` почти наверняка падает, потому что корневой `test` запускает `pnpm -r test`, а в `@tms/web` и `@tms/shared` нет `test`-скрипта. citeturn3view0turn49view0turn23view0  
2) Workflow `.github/workflows/ci.yml` визуально имеет поломанную YAML-структуру (ключи идут на одном уровне без нужной вложенности, например `push` и `branches` выглядят “сиблингами”). Это может привести к тому, что CI вообще не загрузится GitHub Actions или будет интерпретирован неверно. citeturn15view0  

### Корень: workspace, scripts, типичные точки падения

- В корневом `package.json` есть `packageManager: "pnpm@9.15.2"` — хорошо, это фиксирует версию pnpm. citeturn3view0  
- Но в Dockerfile’ах вы используете `corepack prepare pnpm@latest --activate`, что может “тихо” поменять major/minor и сломать сборку/локфайл. Это риск воспроизводимости. citeturn21view0turn48view0turn3view0  
- В обоих Dockerfile используется `pnpm install` без `--frozen-lockfile`: сборка в контейнере может резолвить зависимости иначе, чем CI. citeturn21view0turn48view0turn15view0

## Docker, Compose и деплой

### Compose-файлы: валидность и безопасность

В обоих compose-файлах (`docker-compose.yml`, `docker-compose.prod.yml`) форматирование выглядит “сломанным” (много пустых строк и, главное, сомнительная вложенность/indentation у полей сервисов). Это должно быть зафиксировано как проблема формата и проверено командой:

```bash
docker compose -f docker-compose.yml config
docker compose -f docker-compose.prod.yml config
```

Сейчас, по виду блоков, есть риск, что `docker compose` вообще откажется парсить YAML либо интерпретирует ключи на неверном уровне. citeturn13view0turn14view0  

### Dev compose: пароли и несостыковки env

В dev compose поднимаются Postgres и Redis, причём Redis запускается с `--requirepass ${REDIS_PASSWORD:-tms}`. citeturn13view0  
При этом корневой `.env.example` задаёт `REDIS_URL=redis://redis:6379` без пароля. citeturn47view0  

Это означает риск “dev окружение не поднимается из коробки”: если приложение берёт `REDIS_URL` без пароля, а Redis требует пароль, соединение упадёт и воркеры/очереди не заведутся. Даже если часть кода это обходит, это архитектурно грязная несостыковка, которую нужно устранить на уровне стандарта конфигурации. citeturn13view0turn47view0turn30view0  

### Prod compose: открытые порты БД/Redis (опасно)

В `docker-compose.prod.yml` порт Postgres проброшен на хост (`"5432:5432"`), и порт Redis тоже проброшен наружу (`"6379:6379"`). citeturn14view0  

Для прод-сценария это почти всегда недопустимо: БД и Redis должны быть доступны только во внутренней сети docker/compose, без публикации в интернет.

### Деплой-скрипт: безопасность и идемпотентность

`deploy.sh` ориентирован на один конкретный VPS (в комментарии указан IP и окружение, включая “Server: 5.42.102.58 (Timeweb Cloud, Ubuntu 24.04)”). citeturn50view1turn7view0  
Здесь есть несколько проблем:

- В скрипте используется установка Docker через `curl ... | sh`. Это распространено, но с security/аудит-точки зрения это “remote code execution from internet” без пиннинга версии/хэша. citeturn50view1  
- `.env` генерируется автоматически, что хорошо (случайные секреты), но в нём прямо прописываются `CORS_ORIGIN` и `NEXT_PUBLIC_API_URL` на http + IP. Это означает отсутствие TLS и потенциально опасную модель экспонирования сервисов напрямую на портах 3000/4000. citeturn50view1turn47view0  
- Скрипт делает `docker compose ... build` и `up -d`, но я не подтвердил (по доступным фрагментам), что там есть обязательные шаги миграции/seed; это нужно проверить поиском `db:migrate`/`db:seed` внутри `deploy.sh`. citeturn50view1turn19view0  

### Как поднять dev с нуля

Ниже — сценарий, который должен стать “правдой” после исправления compose/YAML и выравнивания env:

```bash
# 1) Node/pnpm
corepack enable
corepack prepare pnpm@9.15.2 --activate

# 2) Install
pnpm install --frozen-lockfile

# 3) Validate compose
docker compose -f docker-compose.yml config

# 4) Start infra
docker compose -f docker-compose.yml up -d postgres redis

# 5) Migrate DB (из apps/api)
pnpm --filter @tms/api db:migrate

# 6) Seed (только dev)
SEED_PASSWORD='dev-password' pnpm --filter @tms/api db:seed

# 7) Run API+WEB
pnpm dev
```

Обоснование шагов: наличие `db:migrate/db:seed` скриптов в `apps/api`, наличие корневого `dev`, наличие Postgres/Redis в compose и наличие `SEED_PASSWORD` требования в seed. citeturn19view0turn36view0turn3view0turn13view0  

### Как поднять prod безопасно

Минимальный безопасный baseline для прод:

1) Убрать публикацию портов Postgres/Redis на хост в `docker-compose.prod.yml` (оставить их только внутри сети). citeturn14view0  
2) Публиковать наружу только reverse-proxy (80/443), а web/api держать без прямого выхода в интернет (или хотя бы ограничить firewall).  
3) Включить TLS, и в API выставить cookie `secure: true` в прод-конфигурации. Сейчас `secure` зависит от `HTTPS === 'true'`, что легко забыть. citeturn39view2turn50view1  
4) Явно прогонять миграции как часть деплоя (однократно на релиз): `docker compose exec api pnpm --filter @tms/api db:migrate`. citeturn19view0turn21view0  
5) Seed выполнять только в dev/test. В прод — отдельный bootstrap (создание первого admin) с одноразовым токеном. Сейчас seed создаёт множество пользователей. citeturn36view0

Сейчас у вас минимум три источника истины:

- `.env.example` в корне (ориентирован на compose/prod-подобное окружение). citeturn47view0

## Security audit

Ниже — Top-10 рисков безопасности и практические фиксы. Я указываю severity (CRIT/HIGH/MED/LOW) и точку в коде/конфигурации.

### Top-10 рисков

CRIT — БД и Redis опубликованы наружу в prod compose  
Проблема: `docker-compose.prod.yml` пробрасывает Postgres (5432) и Redis (6379) на хост. citeturn14view0  
Impact: удалённый доступ к БД/Redis при ошибке firewall/паролей; Redis часто становится “точкой RCE” (через загрузку модулей/конфигов) в случае доступа.  
Fix: убрать `ports:` у Postgres/Redis в prod compose, использовать только internal network.

CRIT — Небезопасные дефолтные пароли в compose  
Проблема: в compose используются значения по умолчанию `:-tms` для Postgres и Redis. citeturn13view0turn14view0  
Impact: если кто-то поднимет прод без `.env` или с неполным `.env`, сервисы стартуют с предсказуемыми секретами.  
Fix: в `docker-compose.prod.yml` убрать дефолты для секретов вообще; сделать fail-fast.

HIGH — Несостыковка Redis auth (dev) vs REDIS_URL (env)  
Проблема: dev compose включает `--requirepass`, но `.env.example` задаёт `REDIS_URL` без пароля. citeturn13view0turn47view0  
Impact: “не поднимается с нуля”; фоновые задачи и синхронизация ломаются/отключаются (у вас уже есть ветка “Redis unavailable — workers disabled”). citeturn30view0  
Fix: либо убрать пароль из dev Redis, либо добавить пароль в `REDIS_URL` и синхронизировать все окружения.

HIGH — Admin сейчас имеет полный доступ, включая медданные  
Проблема: в RBAC `admin` получает `can('manage','all')`, а ограничения на медданные закомментированы как TODO. citeturn40view2  
Impact: нарушение принципа минимальных прав; потенциальное несоответствие требованиям по медданным (в ТЗ подчёркнута особая категория персональных данных и ограничения ролей). citeturn51view6  
Fix: убрать “полный доступ” даже для admin в прод-конфиге; включить `cannot('read','MedInspectionDetails')` и подобные правила; добавить тесты на RBAC.

HIGH — Append-only триггер для `events` допускает изменение части полей  
Проблема: триггер разрешает UPDATE при условии, что `conflict` изменён и неизменны только некоторые поля; при этом `author_role`, `entity_type`, `version`, `offline_created_at` в условии не зафиксированы, значит могут быть изменены вместе с `conflict`. citeturn33view0turn35view0  
Impact: нарушение неизменяемости event log; это ломает аудит/конфликт-резолвинг (а append-only ключевой в ТЗ). citeturn51view1turn51view0  
Fix: заменить условие на сравнение “все поля кроме conflict неизменны”, например через `to_jsonb(NEW) - 'conflict' = to_jsonb(OLD) - 'conflict'`.

HIGH — Применение trigger SQL на старте API через `unsafe()`  
Проблема: сервер при старте выполняет `sql.unsafe(APPEND_ONLY_TRIGGER_SQL)` до начала слушания порта. citeturn30view4turn33view0  
Impact: если таблицы/расширения/права не готовы — API вообще не запускается; это делает деплой хрупким и может привести к downtime.  
Fix: переносить управление схемой/триггерами в миграции/инициализацию БД (drizzle migrations), а не в runtime start.

MED — Cookie `secure` зависит от `HTTPS === 'true'`  
Проблема: в login cookie `secure` включается только если `NODE_ENV=production` и `HTTPS='true'`. citeturn39view2turn39view1  
Impact: можно “случайно” запустить прод без `HTTPS=true` и получить cookie без secure-флага.  
Fix: в прод всегда ставить `secure: true` (если TLS терминируется на reverse proxy — всё равно cookie должна быть secure).

MED — Секреты/конфиг хостинга и комплаенса 152-ФЗ не формализованы  
ТЗ явно выделяет медданные и комплаенс (152-ФЗ), а также упоминает облака и требования. citeturn51view6turn51view5  
В репозитории есть таблица `med_access_log` и триггер на неизменяемость, но без end-to-end политики secret management (compose secrets, vault) и контроля доступа на уровне окружения это останется “декларацией”. citeturn35view0turn33view0turn47view0  

MED — Логи в pretty-формате вместо структурированных JSON  
Проблема: Fastify logger настроен на `pino-pretty` транспорт. citeturn30view4  
Impact: в проде сложнее собирать логи в Loki/ELK; выше overhead.  
Fix: включать pretty только локально.

## Архитектура домена TMS

### Сопоставление с ТЗ

ТЗ делает append-only и оффлайн-события центральной частью системы: “локальный журнал событий… передаются на сервер без потерь (append-only)… конфликтные пары попадают в очередь ручного разбора”. citeturn51view0  
Также ТЗ фиксирует стек: TypeScript/Node.js(Fastify), Zod, CASL/RBAC, BullMQ(Redis), append-only журнал в PostgreSQL. citeturn51view5  

В коде и структуре проекта это отражено:

- Есть таблица `events` с полями `author_id`, `author_role`, `event_type`, `entity_type`, `entity_id`, `data`, `version`, `conflict`. citeturn35view0  
- Есть Zod-схема события в shared (`EventSchema`) с аналогичными полями, включая `version` и `conflict`. citeturn26view1  
- Есть SQL триггеры, которые пытаются запретить UPDATE/DELETE на `events` (append-only), а также сделать неизменяемыми таблицы осмотров и лог доступа к медданным. citeturn33view0turn34view1turn34view2turn34view3  
- Есть модульная маршрутизация API по доменам (orders/trips/inspections/waybills/fleet/repairs/finance/sync/geo) и единый префикс `/api`. citeturn41view0turn30view5  
- Есть RBAC с выделением медданных как отдельного subject (`MedInspectionDetails`) и явными запретами для диспетчера. citeturn40view4turn51view6  

### Где реализация расходится с ТЗ

- В ТЗ для событий указан `id (UUID v7)` и подчёркнута идемпотентность (“повторная отправка события с тем же id игнорируется”). citeturn51view1  
  В БД сейчас `id` для `events` — `defaultRandom()`, что обычно означает случайный UUID (не v7). citeturn35view0  
  Идемпотентность на уровне API/insert-логики в рамках доступных материалов я не подтвердил (нужно проверить код записи событий и обработку `ON CONFLICT DO NOTHING`). citeturn35view0turn33view0  
- Append-only enforcement для `events` реализован неполно (см. Security audit: можно изменить часть полей вместе с `conflict`). Это прямо подрывает требование append-only. citeturn33view0turn51view1  
- В ТЗ присутствует требование очередей BullMQ (Redis) для фоновых задач. В API видно, что воркеры стартуют при наличии Redis, но детали “retry, DLQ, мониторинг” в рамках этого аудита не подтверждены (нужно смотреть `apps/api/src/integrations/queues.js` и `workers/*`). citeturn30view0turn51view5  

### Фундаментальные компоненты, которые стоит подтвердить или добавить

Не “придумываю”, а формулирую как проверочные гипотезы/дельты:

- Идемпотентность при приёме событий (обязательное требование ТЗ) — проверить обработчик, который пишет в `events`. Должен быть либо `ON CONFLICT DO NOTHING`, либо отдельная таблица дедупликации. citeturn51view1turn35view0  
- Проекции/агрегаты: `orders/trips/...` могут храниться как текущие таблицы + события для аудита. Это допустимо, но тогда критично определить: что является источником истины (таблица состояния или event log). По текущему коду видно, что `events` существует как журнал, но не видно (в рамках этого среза) архитектурного разделения “append-only log → projections”. citeturn35view0turn33view0turn41view0  

## База данных и миграции

### Схема БД зрелая или набросок?

По объёму и наличию индексов это уже больше, чем “набросок”: описано много доменных сущностей (users, contractors, contracts/tariffs, vehicles, drivers, orders, trips, route points, inspections, invoices, restriction zones, events, med audit log и т.д.), есть индексация по ключевым полям (включая `events` и `med_access_log`). citeturn34view4turn35view0  

При этом есть несколько важных архитектурных несоответствий/рисков.

### Конкретные замечания по Drizzle schema (выжимка)

CRIT/HIGH:

1) `events.id` не соответствует “UUID v7” из ТЗ (сейчас `defaultRandom`). citeturn51view1turn35view0  
2) Append-only триггер для `events` допускает изменение не всех полей (см. Security audit). citeturn33view0turn35view0turn51view1  

MED:

3) Денежные поля и тарифы во многих местах используют `real` (float). Для финансовых расчётов это приведёт к ошибкам округления; лучше `numeric(12,2)` или хранение в integer “копейках”. Пример: `tariffs.ratePerKm: real`, суммы в invoices/fines тоже `real`. citeturn34view4turn35view0  
4) Координаты адресов `lat/lon` заданы как `real`. Для гео лучше `double precision` (или PostGIS геотипы), иначе будет теряться точность и появятся артефакты на маршрутизации/геозонах. citeturn35view0  
5) Поля `updatedAt` имеют `defaultNow()`, но без триггера или логики обновления их легко “забыть” обновлять при UPDATE. Например, в `repair_requests` есть `updatedAt` с defaultNow. citeturn35view0  

### Миграции в Docker/prod

- В `apps/api` есть `drizzle.config.ts`, который ожидает `DATABASE_URL` и пишет миграции в `./drizzle`. citeturn20view1  
- В `apps/api/package.json` присутствуют `db:generate` и `db:migrate` через drizzle-kit. citeturn19view0  

Но runtime API не выполняет миграции автоматически, и при этом выполняет триггеры на старте. Это означает, что “поднять с нуля” будет очень зависеть от правильного порядка: сначала миграции/создание таблиц, потом запуск API. citeturn30view4turn33view0turn19view0
