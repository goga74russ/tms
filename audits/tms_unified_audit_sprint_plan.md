# Единый Мастер-Аудит и Пошаговый План Развития TMS (Спринты)

**Дата:** 13 марта 2026  
**Статус проекта:** Post-MVP / Pre-Production (Оценка зрелости бизнес-логики: 40%)

---

## 1. Executive Summary

Проект TMS обладает качественным системным ядром: схемы базы данных (Drizzle ORM), защита транзакций, append-only журнал событий и калькулятор тарифов (с 7 модификаторами) реализованы на высоком уровне. Архитектура готова к масштабированию.

Однако система страдает от **"мнимой готовности" (Fake Readiness)**. Визуально все сущности (заявки, рейсы, путевые листы, инвойсы) существуют, но **сквозные бизнес-процессы между ними разорваны**. Базовые бизнес-инварианты транспортной компании нарушены:
- Финансовый контур (биллинг) работает в отрыве от физического возврата оригиналов документов (первички).
- Исполнение рейса не синхронизировано со статусами клиентских заявок.
- Отсутствует финансовая интеграция штрафов, удержаний и претензий.

Без исправления этих разрывов запуск системы в Production приведет к постоянным кассовым разрывам, жалобам клиентов на неверную тарификацию и юридическим рискам (усталые водители, недействительные путевые листы).

## 2. Топ-10 Критических Бизнес-Рисков (Блокеры запуска)

| ID | Зона | Суть проблемы / Риск | Бизнес-последствие | Приоритет |
| :--- | :--- | :--- | :--- | :--- |
| **1** | Финансы | Счета генерируются по факту завершения `trips`, без контроля возврата оригиналов ТрН. | Отказы в оплате от клиентов, зависшая дебиторка. | 🔥 P0 |
| **2** | Финансы | В тарификаторе (`TarificationService`) ошибки: берутся только первые заявки из массива `trip_orders`, используются несуществующие поля схемы (`nightCoefficient` вместо `nightMultiplier`, `plannedArrivalAt` вместо `plannedCompletionAt`). | Прямые финансовые потери (ошибочно заниженные счета). | 🔥 P0 |
| **3** | Статусы | Завершение рейса (Trip -> `completed`) оставляет связанные заявки (Orders) в статусе `in_transit`. | Хаос в дашборде клиентов, ложное отображение местоположения. | 🔥 P0 |
| **4** | Статусы | Отмена рейса (Trip -> `cancelled`) не освобождает (статус `available`) транспортное средство и прицеп. | Искусственный дефицит автопарка. | 🔥 P0 |
| **5** | Безопасность | API позволяет напрямую менять статус рейса на `waybill_issued` без реальной генерации путевого листа с печатями медика/механика. | Юридическая уязвимость, выпуск неисправных ТС. | 🔥 P0 |
| **6** | Логистика | Разрешено закрытие рейса (`closeWaybill`) с одометром возврата (`odometerIn`), который меньше одометра выезда (`odometerOut`), а также без прохождения всех точек разгрузки. | Махинации с ГСМ. | Высокий P1 |
| **7** | Претензии | Штрафы БД (`fines`) и порча/брак (Инциденты) — это просто текстовые записи, они не вычитаются из счетов клиентам или зарплаты/удержаний водителей. | Прямые убытки ТК (заплатили ЗП водителю, но клиент недоплатил фрахт). | Высокий P1 |
| **8** | Логистика | Прицепы (Trailers) добавлены в БД, но не привязаны к формированию Рейсов (`trips` không имеет `trailerId`) и проверкам (`assignTrip`). | Разбивка логики сцепки. | Высокий P1 |
| **9** | Безопасность| JWT-токены хранятся в `localStorage` (XSS-уязвимость). Доступ роли `client` по CASL к сущности `Order` установлен как `manage` вместо `read`. | Кража сессий, несанкционированное удаление чужих заявок. | Высокий P1 |
| **10** | Документы | ЭТрН генерируется `etrn-generator.ts`, но не подписывается УКЭП провайдера ЭДО. Адреса в ТТН попадают не в те поля. | Юридическая ничтожность документов. | Средний P2 |

## 3. RoadMap: План доработок по Спринтам

Проекту необходимо приостановить разработку новых "фич" и направить ближайшие 3 спринта на отвердение бизнес-процессов. Бэклог пересобирается следующим образом:

### Sprint 10: Сквозные Процессы и Rapid Fixes (P0-Блокеры)
*Цель: Спасти поток ценности, устранить дыры, через которые "утекают" деньги и ломаются статусы.*
- [ ] **Тарификация мульти-заявок:** Переписать `TarificationService`, чтобы он брал массив `trip_orders`, правильно суммировал вес, и исправить названия переменных-множителей. Устранить двойную связь заказов (`orders.tripId` vs `trip_orders`).
- [ ] **Синхронизация статусов завершения:** В `changeTripStatus(COMPLETED)` добавить триггер обновления связанных `orders` -> `delivered`.
- [ ] **Освобождение активов:** В `changeTripStatus(CANCELLED)` возвращать статус ТС и Водителей в `available`.
- [ ] **Запрет обхода путевого листа:** Добавить Guard в смену статуса на `waybill_issued` (запретить, если запись `waybill` физически отсутствует).
- [ ] **Легитимность допусков:** Жестко привязать прохождение предрейсового медосмотра к конкретному `tripId` (сейчас водитель может пройти осмотр для одного рейса, а уехать в другой).
- [ ] **Документооборот (База):** Добавить в таблицу `trips` поле `original_documents_received` (boolean).
- [ ] **Блокировка Финансов:** В `finance.service.ts` запретить генерацию Invoice для рейсов, у которых `original_documents_received = false`.
- [ ] **RBAC Fix:** Понизить права `client` до `read` для `Order` в `auth/rbac.ts`.

### Sprint 11: Контроль Документов и Жесткие Инварианты (P1)
*Цель: Сделать систему надежной для работы логистов, ограничить "человеческий фактор".*
- [ ] **Модуль `Documents`:** Создать полноценный интерфейс "Архивариуса" — реестр сдачи путевых листов и ТрН. Возможность прикреплять сканы (`documents_scans_attached`).
- [ ] **Валидация маршрута:** Запретить завершение рейса, если маршрутные точки (`routePoints`) не находятся в статусе `completed`.
- [ ] **Валидация одометра:** Блокировать `closeWaybill`, если `odometerIn < odometerOut`.
- [ ] **Сцепки (Тягач + Прицеп):** Добавить `trailerId` в таблицы `trips` и `waybills`. Добавить проверку доступности прицепа в алгоритм `assignTrip()`.
- [ ] **Жёсткие блокировки ТС:** Внедрить авто-статус `maintenance` для ТС при заваленном техосмотре. Строгая блокировка назначения водителя по сроку `medCertExpiry`.
- [ ] **Изоляция 152-ФЗ:** Убрать у роли `admin` права на чтение медицинских деталей (`MedInspectionDetails`).
- [ ] **Целостность БД (FK):** Добавить foreign keys для `repairRequests.inspectionId` и переделать `invoices.tripIds` из JSONB в нормализованную join-таблицу.
- [ ] **Безопасность (Auth):** Перевести механику хранения JWT-токенов с `localStorage` на `httpOnly` cookie. Модификация `ttn-pdf.ts` (вывод корректных реквизитов грузополучателя).

### Sprint 12: Интеграция Удержаний и Внешние Данные (P2)
*Цель: Выстроить единый управленческий баланс предприятия (учет потерь).*
- [ ] **Модуль Претензий (Claims):** Создать Workflow создания инцидентов (порча, опоздание) с автоматическим формированием претензии подрядчику/клиенту.
- [ ] **Финансовая интеграция штрафов:** Связать таблицу `fines` с модулем Финансов (`invoices`). Обеспечить автоматическое создание строк "удержание" при генерации Акта выполненных работ или расчете зарплаты.
- [ ] **Платон / Транспондеры:** Интеграция расходов по платным дорогам в калькуляцию себестоимости рейса в `tarification.service.ts`.
- [ ] **Оптимизация (Performance):** Устранить проблему N+1 запросов в `finance.service.ts:generateInvoices`, использовать пакетный расчет `calculateBatchTripCosts`. Добавить составные индексы в БД.
- [ ] **Журнал Аудита:** Усилить SQL-триггер для таблицы `events`, запретив любые частичные `UPDATE` (кроме поля `conflict`), чтобы гарантировать 100% append-only.
- [ ] **РТО (Тахограф):** Полный парсинг данных телематики `tachographRecords`, модуль расчета остатка часов вождения и блокировки назначений при переработках.

### Sprint 13: Промышленный Корпоративный Стандарт (P3)
*Цель: Автоматизация юридически значимых действий.*
- [ ] **ЭДО и КЭП:** Интеграция ЭТрН/ЭПД с провайдером ЭДО по API (Контур/Сбис/Такском) с использованием УКЭП.
- [ ] **ПЭП для Осмотров:** Внедрение Простой Электронной Подписи (СМС/Пин-код) или интеграция с аппаратно-программными комплексами для медика и механика.
- [ ] **GPS / Телематика:** Переход от заглушек (`WialonMock`) на боевое API реальных провайдеров (Wialon, Omnicomm) для автозакрытия точек (`routePoints`) по геозонам.
