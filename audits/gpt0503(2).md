# Глубокий аудит кода и бизнес‑логики репозитория tms

## Область аудита и ограничения наблюдаемости

Этот отчёт построен **по фактическому коду**, доступному в публичном репозитории (backend Fastify/Drizzle + shared schemas). Критические выводы подкреплены ссылками на конкретные файлы/строки GitHub.

Важные ограничения:

- Я **не выполнял интерактивный pentest** вашего прод/стейдж‑стенда и **не входил** в UI по учётке (даже “временной”), потому что это выходит за рамки аудита кода и может восприниматься как несанкционированный доступ.
- Часть разделов ТЗ (например, сравнение с `TZ_TMS.MD`) в рамках этого прохода **не подтверждена ссылками**, потому что этот файл/контент не был доступен в загруженных источниках в ходе сессии. Поэтому любые “соответствует ТЗ” я формулирую только там, где это видно прямо в коде (например, по event journal и триггерам). citeturn44view0turn40view0
- В репозитории обнаружены признаки “смешения” исходников и скомпилированного JS рядом с TS (например, `auth.ts` и `auth.js` рядом). Это снижает уверенность в том, какой код реально исполняется в конкретной среде. citeturn19view0turn16view0

## Как воспроизвести аудит локально

### Базовые команды проекта

В корневом `package.json` заявлен monorepo через workspaces (`apps/*`, `packages/*`), `node >= 20`, и набор scripts (`dev:api`, `dev:web`, `build`, `lint`, `db:*`). citeturn46view0

Минимальный “серийный” прогон (с оговоркой: в корне scripts проксируются через `npm run --workspace=...`, поэтому лучше зафиксировать один менеджер пакетов — см. замечания ниже):

```bash
# 1) установить зависимости
pnpm install

# 2) собрать всё
pnpm run build

# 3) линт (если настроен внутри workspaces)
pnpm run lint

# 4) тесты (рекурсивно, где есть)
pnpm run test
```

Скрипты миграций/сидов проксируются в workspace `@tms/api`: citeturn46view0

```bash
pnpm run db:generate
pnpm run db:migrate
pnpm run db:seed
```

`apps/api/src/auth/auth.ts` намеренно делает fail-fast, если не задан `JWT_SECRET`. Это правильное поведение для production. citeturn21view3


### Что валидировать в первую очередь

1) **Смешение `.js` и `.ts` в `apps/api/src/**`**: убедиться, что runtime не подхватывает “не тот” файл. В дереве `apps/api/src/auth` одновременно лежат `auth.ts` и `auth.js`. citeturn19view0  
2) **“Append-only” SQL триггеры**: у вас есть SQL‑строка триггеров для `events`, `tech_inspections`, `med_inspections`, `med_access_log`. Важно проверить, кто и когда её применяет (миграции vs startup). citeturn40view0turn44view0  
3) **RLS/тенантность**: в роутах часто написано “фильтрация на уровне query”, но в middleware `requireAbility()` нет проверок на объект/владение — только “роль имеет право на тип сущности”. Это ключевой источник IDOR. citeturn23view0turn22view0

## Транзакционность, идемпотентность и гонки

### Где хорошо

- Смена статуса рейса обновляет несколько сущностей (trip, vehicle, orders) в транзакции, плюс запись события в `events` также в транзакции — это правильная композиция. citeturn30view0  
- Генерация номеров `ORD-YYYY-NNNNN` и `TRP-YYYY-NNNNN` делается через `SELECT ... FOR UPDATE` в транзакции — это снижает риск коллизий при параллельных вставках. citeturn15view0turn28view0  
- `generateWaybill()` “закрывает” инвариант в одной транзакции (waybill + trip + event). citeturn35view3turn35view4  

### Где сломано или рискованно

**CRIT — Несогласованность idempotency в event‑journal относительно комментариев и потребностей sync**

В `recordEvent()` написано: “Идемпотентность: если событие с таким ID уже есть — игнорируется”, но фактически:
- `events.id` генерируется автоматически (`defaultRandom()` в схеме), а `recordEvent()` не принимает заранее заданный ID. citeturn43view0turn44view0  
- Значит, при повторе операции (ретраи клиента/очереди) вы почти гарантированно получите **дубликаты событий** в `events`.  

Impact: разъезжается история, ломаются projection/аналитика, конфликтная синхронизация становится неустойчивой.  
Root cause: нет дизайн‑контракта “командный idempotency key → уникальный event id / уникальный constraint”.  
Fix:  
- Ввести `commandId`/`idempotencyKey` в команды (API) и хранить либо:
  - уникальный ключ в `events` (например, `source_event_id`/`external_id`), либо  
  - отдельную таблицу `processed_commands(command_id primary key, ...)` (особенно для sync).  
- В `recordEvent()` добавить поддержку “внешнего ID” и upsert/ignore on conflict.

How to test:
- интеграционный тест: два одинаковых запроса “confirm order” → в `events` ровно одно событие нужного типа.

**MED — Разрыв атомарности между созданием Trip и привязкой Orders**

`createTrip()` создаёт рейс, затем в отдельной транзакции линкует заказы и создаёт route points; после этого пишет `trip.created` событие уже вне транзакций. citeturn29view0turn30view0  

Impact: возможны “полусозданные” рейсы без привязанных заказов, или наоборот частично созданные точки маршрута при сбоях; аудит может расходиться с фактом.  
Root cause: транзакция не охватывает весь “use case”.  
Fix: объединить `insert trip + linkOrdersToTrip + recordEvent(trip.created)` в **одну** транзакцию (или сделать outbox).  
How to test: искусственно уронить вставку routePoints (например, constraint) и убедиться, что trip не создан / откатывается.

**MED — Таймзона “сегодня” для допусков**

Функции `hasValidTechInspectionToday()` / `hasValidMedInspectionToday()` определяют “сегодня” через `new Date()` и `setHours(0..23)`. citeturn39view0turn36view1  

Impact: около смены даты (особенно если сервер в UTC/контейнере) водитель может получить отказ/разрешение не в те сутки, что ожидает бизнес.  
Fix: 
- Ввести “операционную таймзону” (например, `Europe/Moscow`) и считать границы суток в ней (Luxon/Temporal), либо хранить “shift_date” отдельным полем.  
How to test: unit‑тесты для границ суток при разных TZ.

### Sync‑обработка и гонки

Плюс: есть проверка владения рейсом перед приёмом смены статуса (`verifyTripOwnership`). citeturn49view2turn49view3

## AuthN/AuthZ, тенантность и безопасность

### Реализация AuthN

- JWT на `@fastify/jwt`, cookie‑first, header fallback. citeturn21view3turn21view1  
- `JWT_SECRET` обязателен (fail-fast). citeturn21view3  
- Login rate-limit на 5 запросов/мин на endpoint. citeturn21view1turn21view3  

### Реализация AuthZ (RBAC) и ключевой дефект “нет объектного контроля”

`requireAbility(action, subject)` проверяет только `ability.can(action, subject)`, где `subject` — строковый тип сущности. **Проверок на конкретный объект (tenant/owner) здесь нет.** citeturn23view0

Одновременно в ролях присутствует:

- роль `client` имеет `can('manage', 'Order')`. citeturn23view1  
- в роутах Orders “RLS” для клиента/водителя реализуется вручную (фильтрация contractorId/driverId) только в некоторых endpoint’ах, например list/get. citeturn22view0turn13view0  

**CRIT — IDOR/Privilege Escalation сценарий для client по Orders**

Сценарий (подтверждён кодом):  
1) `client` имеет право `manage Order` по RBAC. citeturn23view1  
2) В `PUT /orders/:id` стоит только `requireAbility('update', 'Order')`, но нет гарантированного `contractorId`‑ограничения внутри update‑операции на уровне middleware/DB. citeturn13view0turn23view0  

Impact: клиент, зная чужой `orderId`, потенциально может менять чужие заявки (классический IDOR).  
Root cause: “ручной RLS” не централизован и не применяется на mutation‑endpoint’ах; CASL используется без условий по объекту.  
Fix (практический):
- Ввести единый guard: `assertOrderAccess(orderId, user)` который:
  - для `driver`: проверяет связь через `trip.driverId`;
  - для `client`: проверяет `order.contractorId == user.contractorId`;
  - для staff‑ролей: по политике (например, доступ к любому в рамках организации).
- Вызывать guard во всех `PUT/POST status/cancel/confirm` по Order.
- Либо перейти на “policy per entity row” в CASL (subject object), но тогда объект нужно загружать и передавать в CASL.  

How to test:
- integration test: логин `clientA`, попытка `PUT /orders/:id` где order принадлежит `clientB` → ожидаем 403.

**HIGH — Роль admin сейчас = полный доступ**

В `rbac.ts` роль `admin` временно настроена как `can('manage', 'all')` “для тестирования”, есть комментарий TODO “перед релизом вернуть ограниченные права”. citeturn17view0turn23view0  

Impact: любой компрометированный admin даст полный контроль над системой, включая медданные/финансы (несмотря на изначальные запреты).  
Fix: вернуть продовые ограничения, как минимум:
- запрет чтения `MedInspectionDetails`, `KPI` и т.п. (как в закомментированном плане);
- отделить “superadmin” от “admin” и выдавать строго ограниченно.

**MED — Медданные не зашифрованы на уровне БД**

В схеме `med_inspections` присутствует комментарий “в production шифровать pgcrypto”. Это означает, что сейчас медицинские данные хранятся “как есть”. citeturn31view1turn39view0  
Impact: повышенные регуляторные/юридические риски (в т.ч. утечки), особенно при доступах DBA/бэкапах.  
Fix: выделить PII/медполя и перевести на шифрование (pgcrypto или app-level envelope encryption), плюс строгая матрица доступов и аудит чтения.
