## 1. УПРАВЛЕНИЕ ЗАКАЗАМИ (Orders)

### Жизненный цикл заказа

Статусная машина определена в `packages/shared/src/enums.ts`, переходы в `ORDER_STATE_TRANSITIONS`:

```
new → confirmed → in_transit → delivered
new → confirmed → cancelled
new → cancelled
in_transit → cancelled  (есть!)
delivered → (конечный)
```

`changeOrderStatus()` в `apps/api/src/modules/orders/service.ts:239` валидирует переходы через `canTransition()`. Логика работает корректно.

### Что работает корректно

- Создание заказа (`createOrder()`, строки 92-160): транзакция с `FOR UPDATE` на генерацию номера — правильно
- Идемпотентность номеров: `SELECT ... FOR UPDATE` в `getNextOrderNumber()` исключает дубликаты
- Валидация переходов статусов: `canTransition()` на строке 21 работает
- Событие в журнале при смене статуса (строка 270): `recordEvent()` внутри транзакции

### Проблемы

**P1.1 — КРИТИЧНО: Заказы не переводятся в 'delivered' при завершении рейса**

`apps/api/src/modules/trips/service.ts`, `changeTripStatus(COMPLETED)` (строки 461-478): при завершении рейса обновляются `vehicles.status = 'available'` и `waybills`, но связанные заказы остаются в статусе `'in_transit'`. Статус `'delivered'` недостижим автоматически — только вручную через PATCH `/orders/:id/status`. Бизнес-инвариант нарушен: завершённый рейс не означает доставленный груз.

**P1.2 — ВЫСОКИЙ: Двойная связь заказ↔рейс**

В схеме одновременно существуют:
- `orders.tripId` — прямой FK в `apps/api/src/db/schema.ts:287`
- `tripOrders` (join table) — `apps/api/src/db/schema.ts:331-340`

`linkOrdersToTrip()` в `trips/service.ts:512-572` заполняет ОБЕ. `getTripById()` (строка 212) читает заказы через `orders.tripId` (старая связь). Аналитика и инвойсы (через `finance.service.ts`) работают через `orders.tripId`. Таблица `trip_orders` создана в Sprint 9 для поддержки many-to-many, но старый FK не удалён. Риск расхождения данных при прямых INSERT.

**P1.3 — СРЕДНИЙ: Поиск заказов неполный**

`getOrders()` в `orders/service.ts:170` ищет только по `loadingAddress`. Не ищет по: грузу, контрагенту, номеру заказа (только фильтр, не поиск). По ТЗ §3.4.2 поиск должен работать по всем значимым полям.

**P1.4 — СРЕДНИЙ: Не проверяется активность договора при создании заказа**

`createOrder()` принимает `contractId` без проверки, что договор активен и не истёк. Можно создать заказ по расторгнутому договору.

**P1.5 — НИЗКИЙ: Нет проверки минимального содержимого рейса**

Нет запрета создания рейса без заказов. `createTrip()` создаёт пустой рейс. По ТЗ §3.5 рейс обязан иметь хотя бы одну заявку.

## 2. УПРАВЛЕНИЕ РЕЙСАМИ (Trips)

### Жизненный цикл рейса

Определён в `packages/shared/src/enums.ts`, `TRIP_STATE_TRANSITIONS`:

```
planned → inspection → waybill_issued → loading → in_transit → completed
planned → cancelled
inspection → cancelled
waybill_issued → loading (НО не → cancelled !)
loading → cancelled
in_transit → completed
in_transit → cancelled
```

### Что работает корректно

- `assignTrip()` в `trips/service.ts:237-418`: проверяет статус ТС, ОСАГО, технический осмотр, грузоподъёмность, категорию прав, специальные разрешения (soft), тахограф (soft), инциденты. Логика полная и корректная.
- Блокировки при наличии активных инцидентов (Sprint 9) — работают
- Sequential processing в `processSyncEvents()` для mobile sync — правильно

### Проблемы

**P2.1 — КРИТИЧНО: Нет проверки Режима Труда и Отдыха (РТО)**

`assignTrip()` в `trips/service.ts` проверяет всё, кроме РТО. Нет запроса к `events` для подсчёта часов за последние 24/168 часов. По ТЗ §3.5.4 назначение водителя, нарушающего РТО, должно быть заблокировано. Штраф по КоАП — до 50 000 руб. за рейс.

**P2.2 — КРИТИЧНО: Статус 'waybill_issued' не имеет пути к 'cancelled'**

`TRIP_STATE_TRANSITIONS` в `enums.ts`:
```typescript
WAYBILL_ISSUED: [TripStatus.LOADING], // только LOADING, нет CANCELLED
```
После выдачи путевого листа рейс нельзя отменить без прямого UPDATE в БД. По ТЗ возможна отмена до начала загрузки.

**P2.3 — ВЫСОКИЙ: GPS — 100% заглушка**

`apps/web/src/app/dispatcher/page.tsx` использует `useVehiclePositions` hook через WebSocket, но сам WebSocket подключается к моку (`WialonMock`). Реального позиционирования нет. По ТЗ §3.6 GPS-мониторинг — обязательная функция. В роадмапе: "GPS = 100% mock, реальная интеграция deferred" (Sprint 6).

**P2.4 — ВЫСОКИЙ: Заказы остаются 'in_transit' после завершения рейса**

(дублируется через trips). При `changeTripStatus(COMPLETED)` нет `UPDATE orders SET status = 'delivered'`.

**P2.5 — СРЕДНИЙ: getTripById читает через устаревшую связь**

`trips/service.ts:212`: `orders` загружаются через `trips.with({ orders: ... })` используя `orders.tripId`. Заказы, добавленные только через `trip_orders` (без обновления `orders.tripId`), не попадут в ответ.

## 3. ПУТЕВЫЕ ЛИСТЫ (Waybills)

### Жизненный цикл

```
(нет) → formed → open → closed
```

### Что работает корректно

- `generateWaybill()` в `waybills/service.ts:68`: проверяет `hasValidTechInspectionToday()`, `hasValidMedInspectionToday()`, `getBlockingIncidentsForWaybill()` перед созданием — правильно
- Идемпотентность: строка 85-96 бросает 409 если путевой лист для рейса уже существует
- PDF-генерация формы Ф.4-П в `documents/waybill-pdf.ts`: корректно заполняет все обязательные поля

### Проблемы

**P3.1 — КРИТИЧНО: Статус 'open' недостижим**

В `waybills/service.ts` метод `openWaybill()` отсутствует. Статус переходит `formed → ??? → closed`. По ТЗ §3.7 диспетчер должен "открыть" путевой лист при выезде ТС. Функция есть в интерфейсе, но отсутствует в бэкенде.

**P3.2 — ВЫСОКИЙ: Нет проверки статуса рейса при создании путевого листа**

`generateWaybill()` не проверяет, что `trip.status === 'inspection'`. Путевой лист можно создать для рейса в статусе `'planned'`, `'loading'`, `'in_transit'` или даже `'completed'`. По ТЗ выдача путевого листа — шаг после прохождения осмотров.

**P3.3 — ВЫСОКИЙ: closeWaybill() не проверяет статус рейса**

`waybills/service.ts:219`: закрытие путевого листа возможно для рейса в любом статусе.

**P3.4 — СРЕДНИЙ: ЭТрН требует consigneeInn, которого нет в схеме**

`etrn-generator.ts`: поле `consigneeInn: string` обязательно в интерфейсе. В схеме `orders` (`schema.ts`) поля `consigneeInn` нет. При вызове генератора нужно передавать данные из `contractors`, но join для этого не предусмотрен в маршруте заказа.

**P3.5 — НИЗКИЙ: Водители-совладельцы (waybillDrivers) без RLS**

Sprint 9: `waybillDrivers` таблица добавлена. В `sprint9/routes.ts` нет проверки, что текущий пользователь — диспетчер или механик. Любой авторизованный пользователь может добавить запись.

## 4. ИНСПЕКЦИИ (Inspections)

### Что работает корректно

- **152-ФЗ**: `createMedInspection()` в `inspections/service.ts:459` блокируется на строке 478 если `!driver.personalDataConsent` — правильно
- `medAccessLog` вставляется в транзакции (строки 512-517) — соответствует требованиям
- Поля медицинских данных (давление, пульс, алкоголь) скрываются от ролей без доступа через RBAC
- `getTechInspectionQueue()` (строка 56): batch запросы, нет N+1 — исправлено в Agent 6

### Проблемы

**P4.1 — ВЫСОКИЙ: Медосмотр не привязан к конкретному рейсу**

`hasValidMedInspectionToday()` проверяет наличие ЛЮБОГО одобренного `pre_trip` осмотра за сегодня для данного водителя — без привязки к конкретному `tripId`. Водитель может пройти медосмотр для одного рейса и поехать в другой.

**P4.2 — ВЫСОКИЙ: Нет push-уведомления при отказе в допуске**

`inspections/service.ts:540-548`: при решении `'rejected'` записывается событие в журнал, но нет уведомления: ни водителю через мобильное приложение (FCM/APNS), ни диспетчеру через WebSocket.

**P4.3 — СРЕДНИЙ: Технический осмотр не блокирует ТС при отказе**

`createTechInspection()` с решением `'rejected'`: нет автоматического перевода `vehicles.status` в `'maintenance'`. ТС остаётся доступным для назначения.

**P4.4 — СРЕДНИЙ: Срок медицинской справки не проверяется при выезде**

`assignTrip()` проверяет срок водительских прав и тахограф, но `medCertExpiry` не проверяется. Водитель с просроченной медсправкой будет назначен без блокировки.

## 5. ФИНАНСЫ И ИНВОЙСЫ (Finance/Invoices)

### Что работает корректно

- Структура тарифов: 5 типов (`per_km`, `per_ton`, `per_hour`, `fixed_route`, `combined`) + 7 модификаторов
- `generateInvoices()` обёрнут в транзакцию
- Базовые калькуляции (км × ставка, тонны × ставка, часы × ставка) математически верны при наличии данных

### Проблемы

**P5.1 — КРИТИЧНО: calculateTripCost() использует несуществующие поля схемы**

`tarification.service.ts`:
- Строка 198: `tripRecord.plannedCompletionAt` — поля НЕТ в `schema.ts`. Есть `plannedArrivalAt`.
- Строка 216: `tripRecord.order?.returnRequired` — поля `returnRequired` НЕТ в `orders` и НЕТ в `trips`.
- Строка 191: `tariff.nightMultiplier` — в схеме тарифов поле называется `nightCoefficient`.

Следствие: `urgentCost` всегда равен 0, `returnCost` всегда равен 0, ночная надбавка игнорируется. Все инвойсы систематически занижены.

**P5.2 — КРИТИЧНО: calculateTripCost() использует одиночную связь order**

`tarification.service.ts:87-109`:
```typescript
db.query.trips.findFirst({ with: { order: ... } })
```
Обращается к `orders.tripId` (один заказ). При рейсе с несколькими заказами через `trip_orders` берётся только первый. Для multi-order рейсов тариф рассчитан неверно.

**P5.3 — ВЫСОКИЙ: generateInvoices() использует устаревшую связь**

`finance.service.ts:53`: запрос через `orders.contractorId` → join к `orders.tripId`. При рейсах с заказами только в `trip_orders` (без `orders.tripId`) заказы не попадут в выборку и инвойс не сформируется.

**P5.4 — ВЫСОКИЙ: invoices.tripIds хранится как JSONB без FK**

`schema.ts:541`: `tripIds: jsonb().$type<string[]>()`. Нет ссылочной целостности. Удалённый рейс не приведёт к каскадному удалению из инвойса.

**P5.5 — ВЫСОКИЙ: calculateBatchTripCosts() молча пропускает рейсы без тарифа**

`tarification.service.ts:352`: `continue` без логирования при отсутствии тарифа для рейса. Инвойс сформируется с нулевой суммой без предупреждения.

**P5.6 — СРЕДНИЙ: KPI топ-водителей содержит захардкоженные данные**

`finance.service.ts:246-250`: `eco: '95%'` — захардкоджен для всех водителей.

**P5.7 — СРЕДНИЙ: N+1 в generateInvoices()**

`finance.service.ts:77`: `calculateTripCost()` вызывается в цикле для каждого рейса. Хотя создан `calculateBatchTripCosts()` (строка 304), в `generateInvoices()` он не используется. Числится как C-3 в audit-log.

## 9. АВТОРИЗАЦИЯ И RBAC (Auth/RBAC)

### Что работает корректно

- JWT-авторизация через Fastify plugin на всех маршрутах
- CASL AbilityBuilder для 10 ролей в `rbac.ts`
- `admin`: `can('manage', 'all')` + `cannot('read', 'MedInspectionDetails')` — семантика CASL правильная
- `assertOrderAccess()` в `guards.ts`: проверяет staff/client/driver — работает

### Проблемы

**P9.1 — КРИТИЧНО: JWT в localStorage — уязвимость XSS**

`waybills/page.tsx` и `finance/page.tsx`: `downloadPdfAuth()` читает токен из `localStorage`. Весь фронтенд хранит JWT в localStorage. Любой XSS-скрипт получает токен. По ТЗ §5.6 требуется httpOnly cookie + CSRF-токен. Открытый issue H-15.

**P9.2 — ВЫСОКИЙ: client может 'manage' Order**

`rbac.ts:163-170`: `can('manage', 'Order')`. CASL `'manage'` = create, read, update, delete. По ТЗ Sprint 9 §3.17 клиентский портал предоставляет только read-only доступ к своим заказам.

**P9.3 — ВЫСОКИЙ: Нет PostgreSQL Row-Level Security**

RLS реализован только на уровне приложения. Прямой доступ к БД (pgAdmin, Drizzle Studio) не блокирует видимость чужих записей. Открытый issue H-3.

**P9.4 — ВЫСОКИЙ: Zod `as any` в 17 обработчиках**

`fleet/routes.ts`, `repairs/routes.ts`, `trips/routes.ts`: `request.body as any` без валидации схемы. Риск инъекции произвольных полей в Drizzle-запрос. Открытый issue H-4.

**P9.5 — СРЕДНИЙ: accountant имеет 'manage' Contractor**

`rbac.ts:144`: бухгалтер может создавать и удалять контрагентов. По ТЗ это функция логиста/менеджера.

**P9.6 — НИЗКИЙ: ПЭП без серверной верификации**

Простая электронная подпись в мобильном приложении (signature base64) не верифицируется на сервере. `routePoints.signatureUrl` хранит URL изображения без криптографической проверки.

## 10. СИНХРОНИЗАЦИЯ И ИНТЕГРАЦИИ (Sync/Integrations)

### Что работает корректно

- `processSyncEvents()` в `sync/service.ts:32`: sequential processing для сохранения порядка событий
- `verifyTripOwnership()` (строки 19-30): проверка принадлежности рейса водителю
- Conflict detection: если рейс `'cancelled'` на сервере — offline-событие отклоняется с явной ошибкой
- Идемпотентность через `externalId` в журнале событий

### Проблемы

**P10.1 — КРИТИЧНО: GPS — 100% заглушка без пути к продакшену**

`dispatcher/page.tsx`: интеграция с Wialon существует только как `WialonMock`. Нет реального WebSocket endpoint к GPS-провайдеру. По ТЗ §3.6 GPS-мониторинг — ключевая функция.

**P10.2 — ВЫСОКИЙ: ЭПД (ЭТрН) — прототип без ФНС-сертификации**

`etrn-generator.ts`: XML-генерация по приказу ЕД-7-26/383@. Нет: подписи УКЭП, интеграции с ФНС-шлюзом, тест-стенда ФНС. По ТЗ §3.16 ЭПД обязателен с сентября 2026 (4 месяца). Риск высокий.

**P10.3 — ВЫСОКИЙ: 1С-интеграция отсутствует полностью**

По ТЗ §3.15 требуется синхронизация с 1С:Предприятие для контрагентов и договоров. В codebase нет ни одного файла с 1С-интеграцией. В audit-log не числится как задача.

**P10.4 — СРЕДНИЙ: WatermelonDB sync — нет rate limiting**

`/sync/push` принимает массив событий без ограничения размера. Нет `max batch size`.

**P10.5 — НИЗКИЙ: recordEvent вне транзакции**

В некоторых местах `recordEvent()` вызывается после `db.transaction()`, а не внутри. Если `recordEvent` упал — событие не записано, но данные изменены. Числится как N-2 в audit-log.

## 11. ДОКУМЕНТЫ И PDF (Documents/PDF)

### Что работает корректно

- `waybill-pdf.ts`: форма Ф.4-П корректно заполняется с механиком, медиком, водителем, маршрутом, одометром
- TTN (форма 1-Т) в `orders/routes.ts:142-233`: отдельный endpoint, PDF генерируется
- `streamToBuffer(doc)` корректно конвертирует PDFKit-стрим в Buffer для отдачи через HTTP
- Кириллица через Roboto TTF (Roboto-Regular.ttf + Roboto-Bold.ttf)
- Браузерная печать: 4 print-страницы (`/print/waybill/`, `/print/invoice/`, `/print/act/`, `/print/ttn/`) с `window.print()` и `@media print` CSS

### Проблемы

**P11.1 — ВЫСОКИЙ: TTN использует адрес вместо наименования грузополучателя**

`orders/routes.ts:231`: `consigneeName: order.unloadingAddress ?? '—'`. В форме 1-Т поле "Грузополучатель" должно содержать наименование юридического лица, не адрес. ТТН с некорректными данными юридически ничтожна.

**P11.2 — ВЫСОКИЙ: ЭТрН требует consigneeInn, которого нет в заказах**

`etrn-generator.ts`: поле `consigneeInn` обязательно. В `orders` нет колонки `consigneeInn`. Нужен join к `contractors`, но не реализован.

**P11.3 — СРЕДНИЙ: Нет endpoint для PDF акта выполненных работ**

`finance/routes.ts`: нет endpoint `/finance/invoices/:id/pdf-act`. По ТЗ §3.9 акт — обязательный документ при закрытии инвойса.

**P11.4 — СРЕДНИЙ: S3 не реализован — файлы на локальном хранилище**

Фотографии и подписи сохраняются локально. При перезапуске контейнера файлы теряются.

**P11.5 — НИЗКИЙ: CARRIER константа захардкожена в pdf-base.ts**

Данные организации (ИНН, наименование) зашиты в код, не берутся из БД. При смене организации требуется перекомпиляция.

## 12. ПРОИЗВОДИТЕЛЬНОСТЬ И МАСШТАБИРОВАНИЕ

### Что работает корректно

- `FOR UPDATE` при генерации номеров документов: исключает race condition
- `db.transaction()` используется в критичных местах
- Redis + BullMQ: фоновые задачи вынесены в очереди
- `getTechInspectionQueue()`: исправлен N+1 (Agent 6, JOIN вместо 3 SELECT)

### Проблемы

**P12.1 — ВЫСОКИЙ: N+1 в generateInvoices() — не исправлен**

`finance.service.ts:77`: `calculateTripCost()` вызывается в цикле. При 1000 рейсов в инвойсе — 1000 SELECT запросов. Числится как C-3 в audit-log.

**P12.2 — ВЫСОКИЙ: getOrders() без полноценной пагинации**

`logist/page.tsx`: `fetch('/api/orders?limit=100')`. Нет cursor-based pagination. При росте данных производительность деградирует.

**P12.3 — СРЕДНИЙ: Нет кэширования справочников**

Список ТС, водителей, тарифов запрашивается при каждом открытии формы. Redis есть в инфраструктуре, но кэш справочников не реализован.

**P12.4 — СРЕДНИЙ: Отсутствуют составные индексы**

В `schema.ts` нет индексов для:
- `trips` по `(status, driverId)` — фильтрация активных рейсов водителя
- `orders` по `(status, contractorId)` — фильтрация заказов контрагента
- `events` по `(entityType, entityId, createdAt)` — запросы истории

**P12.5 — НИЗКИЙ: WatermelonDB pull — нет delta compression**

При крупной дельте (водитель offline неделю) трафик может быть значительным.
