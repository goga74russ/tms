# Полный аудит бизнес‑логики TMS проекта tms

## Executive Summary

Проект — это попытка собрать «операционную» TMS для компании с собственным автопарком с упором на предрейсовый допуск (механик/медик), путевой лист, базовые рейсы/заявки, ремонтный контур и первичный контур финансов (расчёт рейса, счета/акты/УПД, выгрузка в 1С). В репозитории явно присутствует доменная структура на уровне API‑модулей (orders/trips/inspections/waybills/repairs/finance/sync/fleet/geo) и единый «журнал событий», что является хорошим фундаментом для аудита и доказуемости операций. citeturn8view0turn14view0turn19view0turn23view0turn25view0turn33view0turn29view0turn39view0

Уровень зрелости по бизнес‑логике: **средний для демо/учебного контура**, но **ниже порога промышленной TMS**, если критерием считать «невозможность обхода обязательных этапов» и «сквозное закрытие перевозки документами/финансами без ручных “костылей”». Ключевая причина — **несостыкованность статусов/процессов между модулями** (рейс ↔ путевой лист ↔ осмотры ↔ закрытие ↔ биллинг) и **наличие обходных путей** (через универсальные эндпоинты смены статуса рейса), а также **разрывы доменных контуров** (претензии/расхождения/возврат оригиналов/инциденты как управляемый жизненный цикл). citeturn21view0turn25view0turn29view0turn46view0turn47view1turn41view0

Главный сильный аспект: **предрейсовые осмотры + выпуск путевого листа как “точка допуска”**, где действительно реализована строгая бизнес‑проверка наличия **двух допусков за текущий день** (тех + мед) и блокировка при «blocking incidents». citeturn25view0

Главный риск: **“фейковая готовность” за счёт UI/статусов/ролей/таблиц**, когда выглядит как TMS, но в реальной эксплуатации возникают ситуации: «рейс переведён в нужный статус без путевого листа», «отмена рейса не освобождает ТС», «расчёт/биллинг не выдерживает мультизаявочности» и «процессы спорных случаев/претензий не поддержаны». citeturn21view0turn20view0turn29view1turn46view0turn8view0

## Что реально реализовано по бизнес‑логике

В этом разделе я перечисляю **не “страницы/компоненты”**, а именно **доменное поведение**, которое можно считать работающим сквозным фрагментом процесса.

**Заявки клиента (Orders)**  
Есть полноценная CRUD‑логика заявки с авто‑нумерацией, фильтрами, Kanban‑представлением по статусам и базовой state machine переходов статусов. Переходы валидируются функцией `canTransition`, а изменения сопровождаются записью событий `recordEvent`. Есть назначение заявки на рейс (требуется статус `confirmed`). Также присутствует генерация PDF «ТТН» для заявки (маршрут `/orders/:id/ttn`) с подмешиванием данных ТС/водителя/путевого/рейса при наличии связей. citeturn14view0turn16view0

**Рейсы (Trips) и планирование маршрута**  
Рейс создаётся в статусе `planning`, допускает привязку множества заявок через `orderIds`. При линковке заявок автоматически создаются точки маршрута (loading/unloading) с очередностью и окнами. Статусы рейса описаны state machine (planning→assigned→inspection→waybill_issued→loading→in_transit→completed→billed, плюс cancelled). Есть выдача списков доступных ТС/водителей, причём водители фильтруются по занятости (в активных рейсах). citeturn19view0turn20view0turn21view0

**Назначение ТС и водителя с проверками ограничений (частично “hard”)**  
Функция `assignTrip` реализует набор проверок перед назначением:
- доступность ТС (статус `available`);
- просрочка ключевых документов ТС (техосмотр, ОСАГО);
- грузоподъёмность (суммарный вес грузов заявок рейса vs capacity ТС);
- активность водителя и срок действия ВУ;
- «мягкие» предупреждения про отсутствие пропусков и просрочку калибровки тахографа/медсправки;
- блокировка при открытых/расследуемых инцидентах “blocksRelease”.  
При наличии hard‑блоков назначение не выполняется и API возвращает 409. citeturn19view0turn20view0turn21view0

**Предрейсовые осмотры (Tech + Med) с юридическими ограничениями и неизменяемостью**  
Техосмотр:
- есть очередь «кому сегодня надо пройти», завязанная на рейсы со статусом `assigned` и отсутствием approved‑техосмотра по ТС за сутки;
- есть чек‑лист‑шаблоны;
- есть ПЭП/подпись (`signature`);
- при отказе автоматически создаётся заявка на ремонт и переводится статус ТС в `broken`. citeturn23view0turn24view0turn33view0  

Медосмотр:
- есть очередь ожидания;
- есть контроль согласия на обработку ПДн (152‑ФЗ) как блокирующее условие выполнения медосмотра;
- ведётся аудит доступа к медданным (`medAccessLog`);
- реализовано разграничение «медик видит детали, остальные — только факт допуска/результат». citeturn23view0turn24view0turn41view0  

Также предусмотрены SQL‑триггеры, запрещающие UPDATE/DELETE для журнала событий и журналов осмотров/доступа (append‑only), с единственным разрешённым исключением для пометки `conflict` у событий. citeturn51view2turn39view0

**Путевые листы (Waybills) как “точка выпуска на линию”**  
Критически важная часть реализована: генерация путевого листа для рейса требует **наличия двух допусков за текущий день** (техосмотр по ТС + медосмотр по водителю) и проверяет блокирующие инциденты. Выпуск путевого листа идемпотентен по рейсу (не даёт создать второй). При формировании:
- создаётся путевой лист с номером WB‑YYYY‑NNNNN;
- рейсу проставляется `waybillId` и статус `waybill_issued`;
- фиксируется одометр “на выезд”;  
Есть закрытие путевого листа с одометром “на въезд” и обновлением одометра ТС, а также PDF путевого листа. Есть генерация XML ЭТрН (титул 1 и 4) как отдельные эндпоинты (скорее как заготовка). citeturn25view0turn26view0

**Ремонтный контур (Repairs)**  
Есть state machine заявки на ремонт (created→waiting_parts→in_progress→done), создание заявки с влиянием на статус ТС (maintenance/broken), возврат ТС в available при завершении ремонта с учётом «нет других активных ремонтов», плюс аналитика и плановое ТО‑триггер (cron‑подразумеваемо). citeturn33view0turn33view1

**Финансы и биллинг (частично, но реально)**  
Имеется:
- расчёт стоимости рейса по тарифу (tarificationService);
- генерация счетов/актов/УПД за период и контрагента, перевод рейсов completed→billed;
- смена статуса счета (draft→sent→paid/overdue);
- PDF счета/акта;
- выгрузка в 1C (JSON legacy + CommerceML XML). citeturn27view1turn29view0turn26view0

**RBAC/RLS и защита от IDOR**  
Роли (logist/dispatcher/driver/medic/mechanic/accountant/admin/client и др.) и доступы описаны CASL‑правилами. Для ключевых сущностей (orders/trips/waybills) предусмотрены централизованные guard‑проверки владения/доступа, чтобы водитель/клиент не мог читать чужие объекты по ID. citeturn41view0turn41view2turn16view0turn21view0turn25view1

**Оффлайн‑синхронизация событий водителя**  
Есть водительский push событий: изменение статуса рейса и статусы точек маршрута (arrived/completed) с проверкой, что пользователь действительно водитель назначенного рейса. Это сильный задел под мобильный контур POD (фото/подпись) и “фактологию” исполнения. citeturn38view0turn38view1turn20view0

## Что заявлено или подразумевается как TMS, но не реализовано или опасно неполно

Здесь собраны разрывы именно **сквозного процесса** и домена перевозчика, а не «не хватает красивой архитектуры».

**Инциденты, претензии, расхождения: контур “спорных случаев” отсутствует как управляемый процесс**  
Таблица инцидентов присутствует в модели данных, инциденты используются как блокирующий фактор для выпуска путевого листа/назначения (логика “blocksRelease”), и права на Incident заложены в RBAC. Но в списке API‑модулей нет отдельного управляемого контура incident‑workflow (создание/расследование/финансовые последствия/закрытие), из‑за чего «расследовать постфактум» и связывать спор с деньгами невозможно на уровне бизнес‑процесса. citeturn8view0turn41view0turn25view0turn19view0turn32view0

**Возврат оригиналов документов и юридически значимый жизненный цикл документов перевозки — не найден**  
Есть генерация PDF/XML документов (ТТН, путевой, счёт/акт/УПД), но в доменной схеме отсутствует контур «реестр оригиналов», статусы “оригинал получен/не получен/утрачен”, маршрутизация по подразделениям и блокировки закрытия перевозки без обязательного пакета. Текущая модель “документ = PDF по запросу/сущность путевого листа” не покрывает жизненный цикл бумажного документооборота перевозчика. citeturn16view0turn25view0turn26view0turn27view1turn55view0turn32view0

**Трейлеры/прицепы формально есть, но не интегрированы в назначение и инварианты**  
В БД и RBAC присутствуют Trailer/WaybillDriver/WaybillExpense, но в базовых потоках назначения рейса/выпуска путевого листа нет проверок сцепки, совместимости, требований заявок и запрета выхода без прицепа/полуприцепа в нужных типах перевозок. citeturn41view0turn32view0turn19view0turn25view0

**РТО/тахографические ограничения не доведены до блокировок (только атрибуты/предупреждения)**  
Есть таблица тахографических записей и “soft warning” про калибровку тахографа, но нет бизнес‑проверок соблюдения режима труда и отдыха водителя как hard‑инварианта назначения/выпуска. Это критично для реального перевозчика. citeturn32view0turn20view0

**Интеграции (Wialon/ГИБДД/DaData/топливные карты) — по сути заглушки/моки**  
Очереди BullMQ и воркеры реализованы, но данные берутся из мок‑источников (WialonMock/GibddMock/DaDataMock/FuelCardMock). Для эксплуатации это означает: “карта/пробег/штрафы/заправки” не являются источником истины и не могут быть основой KPI/биллинга/расследований. citeturn43view0turn45view0turn45view1turn37view0

**Геокодирование/дистанции — также дев‑заглушки**  
Геокодер использует словарь “известных адресов” и детерминированный hash‑mock (координаты “рядом с Москвой”), а планирование расстояния — Haversine × коэффициент. Это не годится для реального расчёта километража/маршрута, контроля зон/окон, платных дорог и т.п. citeturn37view0turn37view1turn36view0

## Сквозной аудит процесса от заявки до закрытия рейса

Ниже — аудит **сквозного бизнес‑флоу** «заявка клиента → рейс → выпуск → выполнение → подтверждение → закрытие → биллинг», с указанием сущностей/статусов/проверок/документов и реальных мест в коде.

### Приём заявки

- Сущности: `orders` (Заявка), `contractors` (Контрагент), опционально `contracts`/`tariffs`. citeturn14view0turn32view0turn35view0  
- Статусы: `draft` по умолчанию. citeturn14view0turn32view0  
- Реальные переходы: есть state machine переходов заявки, но на шаге приёма используется просто создание. citeturn14view0turn16view0  
- Проверки: в API включена Zod‑валидация (`OrderCreateSchema`), авто‑генерация номера, но нет доменных проверок «валидность договора/тарифа/условий перевозки» на уровне бизнес‑инвариантов. citeturn16view0turn14view0  
- Документы: на этом шаге документ не создаётся, но есть возможность позже сгенерировать PDF ТТН. citeturn16view0  
- Риск: заказ можно принять без “юридического контура” договора/условий (операционно допустимо для черновика), но это должно быть закреплено правилами подтверждения. citeturn14view0turn16view0

### Подтверждение заявки

- Статусы: `draft → confirmed`, отдельный эндпоинт `/orders/:id/confirm` и общий `/orders/:id/status`. citeturn16view0turn14view0  
- Проверки: валидируется только допустимость перехода по state machine; доменных “гейтов” (обязательные поля, SLA‑согласование, подтверждение цены) не обнаружено в логике смены статуса. citeturn14view0turn16view0  
- Документы: подтверждение заявки не формирует договор/заказ‑наряд. citeturn16view0

### Планирование рейса

- Сущности: `trips` (Рейс), `route_points` (Точки маршрута), связка заявок через `orders.tripId`. citeturn19view0turn20view0turn32view0  
- Статусы: рейс создаётся со статусом `planning`. citeturn19view0turn32view0  
- Переходы: state machine рейса определена, но создание — “планируется”. citeturn19view0turn21view0  
- Проверки: рейс может быть создан без заявок (orderIds необязателен). Это допустимо лишь если в бизнес‑модели есть “порожние/перегоны”; в ТЗ это должно быть явно, иначе это инвариант‑дырка. citeturn19view0turn21view0  
- Автоматизации: при линковке заявок создаются точки погрузки/выгрузки (pending) с окнами. citeturn20view0turn19view0

### Подбор ТС и водителя

- Сущности: `vehicles`, `drivers`, плюс проверки по `permits` и “blocking incidents”. citeturn20view0turn32view0  
- Статусы: назначение возможно только из `planning`, после назначения рейс становится `assigned`, а ТС получает статус `assigned`. citeturn20view0turn21view0  
- Проверки (hard): доступность ТС, просроченный техосмотр/ОСАГО, перегруз по сумме весов заявок, неактивный водитель, просроченное ВУ, блокирующие инциденты. citeturn20view0turn19view0  
- Проверки (soft): отсутствие пропусков, калибровка тахографа, медсправка. citeturn20view0  
- Чего не хватает для “реальной TMS”: проверки типов кузова/объёма/температурного режима/гидроборта под требования заявки (данные есть в заявке и ТС, но явной проверки при назначении нет). citeturn14view0turn32view0turn20view0turn35view0

### Предрейсовый техосмотр

- Сущности: `tech_inspections`, `checklist_templates`, `repair_requests`. citeturn23view0turn32view0turn33view0  
- Проверки: чек‑лист не пустой, решение approved/rejected, подпись (ПЭП). При rejected создаётся ремонт и переводится статус ТС в `broken`. citeturn24view0turn23view0  
- Гейт/инвариант: сам по себе техосмотр не меняет статус рейса и не “выпускает”, он лишь фиксирует допуск/недопуск и может инициировать ремонт. citeturn23view0

### Предрейсовый медосмотр

- Сущности: `med_inspections`, `med_access_log`, `checklist_templates`. citeturn23view0turn32view0  
- Проверки: обязательно согласие на обработку ПДн как условие проведения медосмотра в системе; события по медосмотру не раскрывают медицинские детали (152‑ФЗ‑подход). citeturn23view0turn24view0turn41view0turn51view2  
- Роль: очередь медосмотров доступна только медику (route‑проверка). citeturn24view0

### Выпуск на линию и формирование путевого листа

- Сущности: `waybills`, связь с `techInspections` и `medInspections`, а также блокирующие `incidents`. citeturn25view0turn32view0  
- Ключевой инвариант реализован: **нельзя сформировать путевой лист без двух допусков сегодня**. citeturn25view0  
- Документы: путевой лист как сущность + PDF, плюс XML ЭТрН (титулы) как export. citeturn26view0turn25view0  
- Опасный обход: в рейсах есть универсальный эндпоинт смены статуса `/trips/:id/status`, который вызывает `changeTripStatus` без доменных “guard’ов” по обязательности путевого листа/допусков. Это означает, что организационно роль “dispatcher” (у которой есть manage Trip) может **проставить `waybill_issued` без фактического формирования путевого листа**, обойдя ключевой инвариант выпуска. citeturn21view0turn20view0turn41view0turn25view0

### Движение рейса по статусам и контроль исполнения

- Статусы рейса: `loading`, `in_transit`, `completed`, `billed` и др. Переходы валидируются state machine. citeturn19view0turn21view0  
- Контроль по точкам: `route_points` имеют статусы `pending/arrived/completed/skipped`, таймстемпы прибытия/завершения, фото и подпись. citeturn20view0turn32view0turn38view1  
- Оффлайн‑факты водителя: driver‑sync может проставлять `arrived/completed` по точкам и менять статус рейса, если водитель действительно назначен на рейс. citeturn38view1turn38view0  
- Не найдено: доменный слой “частичная доставка/отказ/срыв/перенос/простой как инцидент с последствиями” (часть можно эмитировать точками и заметками, но нет формализованного процесса). citeturn20view0turn32view0

### Подтверждение доставки (POD)

- Реализовано частично: подтверждение в модели — это завершение unloading‑точки с `signatureUrl` и `photoUrls`. Это ближе к “электронному POD”, но отсутствует:
  - отдельная сущность “Proof of Delivery / акт приёмки” с подписью контрагента и статусами;
  - правила: нельзя завершить рейс без факта выполненных unloading‑точек и подтверждения. citeturn20view0turn38view1turn19view0

### Закрытие рейса и возврат документов

- Закрытие путевого листа (`closeWaybill`) обновляет одометр ТС и поля рейса (odometerEnd/fuelEnd/actualCompletionAt), но не завершает рейс в статус `completed` и не проверяет “POD/документы”. citeturn25view0  
- Завершение рейса как статус (`TripStatus.COMPLETED`) делается через `changeTripStatus` и может освободить ТС (`available`). citeturn20view0  
- Разрыв: закрытие путевого листа и завершение рейса — два разных механизма без обязательной синхронизации и без жестких “гейтов” комплектности документов. citeturn25view0turn20view0  
- Возврат оригиналов документов в доменном виде не найден (см. выше). citeturn32view0turn55view0

### Биллинг, актирование, УПД

- Генерация счетов/актов/УПД выбирает рейсы со статусом `completed` за период и контрагента и переводит их в `billed`. citeturn29view0turn27view1  
- Риск: логика тарификации опирается на `tripRecord.order` и поля, которые неочевидно присутствуют в текущей модели рейса (например `plannedCompletionAt`, флаги типа `returnRequired`), и **не учитывает мультизаявочность рейса корректно** (вес/условия/маршрут как сумма заявок). citeturn46view0turn47view1turn29view0turn20view0turn32view0

## Нарушения бизнес‑инвариантов, проблемы модели данных, статусов и ролей

Ниже — концентрат «где система допускает невозможные/опасные состояния».

**Рейс можно “выпустить” без путевого листа (обход ключевого гейта допуска)**  
Путевой лист формируется только через `generateWaybill`, где реально проверяются оба допуска. Но есть прямой эндпоинт смены статуса рейса `/trips/:id/status`, который никак не требует наличия `waybillId` или вызова `generateWaybill` при переходе в `waybill_issued`. Это ломает главный операционный инвариант «без механика и медика выпуск невозможен». citeturn25view0turn21view0turn20view0turn41view0

**Отмена рейса не освобождает ТС (зависание ресурса автопарка)**  
В описании маршрута отмены заявлено “освобождение ТС”, но в `changeTripStatus` освобождение ТС явно реализовано при `completed`, а для `cancelled` аналогичного действия не видно. Это ведёт к “вечному assigned” у ТС и невозможности использовать его далее. citeturn21view0turn20view0

**Несогласованность статуса “inspection” между модулями**  
State machine рейса содержит статус `inspection`, но очередь техосмотра/медосмотра формируется по рейсам со статусом `assigned`. Это создаёт путаницу: какой статус означает “ожидает осмотров” и где реально живёт стадия предрейсового допуска. citeturn19view0turn23view0turn24view0

**Модель данных допускает “осиротевшие” связи**  
В схеме рейса есть поля `waybillId`, а в заявке `tripId`, но эти поля не оформлены как строгие FK‑ссылки (в отличие от route_points и waybills). Это повышает риск несогласованных состояний при ошибках/ручных исправлениях/импортах. citeturn32view0

**Тарификация/биллинг конфликтует с мультизаявочностью рейса**  
Рейсы в операционном контуре поддерживают множество заявок (orders.tripId, создание множества route_points), но тарификация использует концепт `tripRecord.order` и ряд “однозаявочных” атрибутов. В результате: стоимость рейса может быть рассчитана по одной заявке при фактическом объединении нескольких, а модификаторы “срочность/возврат” могут вообще не сработать. citeturn20view0turn46view0turn47view1turn29view0turn32view0

**Закрытие путевого листа ≠ закрытие рейса (и ≠ готовность к биллингу)**  
Путевой лист можно закрыть, обновив одометр/время, но рейс может остаться не в `completed`, а биллинг отбирает только `completed`. Это создаёт “операционный тупик”: документы закрыты, пробег зафиксирован, но счет не сформируется. citeturn25view0turn29view0

**Роли и границы ответственности не всегда обеспечивают невозможность обхода чужого этапа**  
RBAC по CASL задаёт права, но ключевое — “workflow guard’ы” (проверки обязательных стадий) — не везде. Пример: dispatcher управляет Trip и может вызвать смену статусов, обходя контур путевого листа. Это не вопрос прав, это вопрос доменных инвариантов. citeturn41view0turn21view0turn25view0

## Риски и план приведения к “реальной TMS”

### Насколько проект годен для реальной эксплуатации

На текущем состоянии систему можно использовать как “учётный контур” для:
- ведения заявок и их базовых статусов;
- планирования рейса и назначения ТС/водителя с частью проверок;
- проведения тех/мед осмотров и формирования путевого листа (в рамках контуров, где запрещён обход);
- фиксации фактов по точкам маршрута (включая оффлайн‑события). citeturn14view0turn20view0turn23view0turn25view0turn38view1  

Но **вести реальные перевозки “как в промышленной TMS” рискованно**, потому что:
- ключевые инварианты выпуска/закрытия можно обойти через общие эндпоинты смены статуса рейса;
- отмена может “застрять” в ресурсе автопарка;
- закрытие перевозки документами и биллинг не являются единым защищённым workflow;
- нет домена претензий/расхождений/возврата оригиналов;
- расчёт стоимости/акты потенциально некорректны для рейсов с несколькими заявками. citeturn21view0turn20view0turn25view0turn29view0turn46view0turn32view0
