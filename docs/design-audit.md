# Design Audit (Web + Mobile UX)

Дата: 6 марта 2026  
Область: `apps/web`, ключевые роли + клиентский портал + mobile UX-принципы

## 1. Executive Summary
Текущий UI уже функционален и роли разделены, но дизайн пока больше “экраны-формы”, чем “операционная система принятия решений”.  
Главный потенциал роста: ускорить диспетчера и логиста через decision-first UX.

## 2. Что уже хорошо
1. Ролевые страницы и базовая навигация есть.
2. Единый визуальный язык на Tailwind + ui-компонентах.
3. Есть карты, таймлайн, канбан, KPI/аналитика.
4. Есть live-статус (WS) и понятные статусные бейджи.

## 3. Ключевые UX gaps
### P1 (критично для эффективности)
1. Нет единого `Control Tower` (риски/действия разнесены по экранам).
2. Слабая explainability решений (почему такой статус/назначение/риск).
3. Мало “следующих действий” (оператору нужно самому догадываться, что делать).
4. Нет плотного режима для диспетчерской работы (high-density table/list mode).

### P2 (важно для внедрения и удержания)
1. Клиентский портал слаб по доверительным метрикам (ETA/SLA/причины задержек).
2. Неполная стандартность error/empty/loading states между экранами.
3. Недостаточно batch действий (массовое подтверждение/назначение/перенос).

### P3 (улучшает perceived quality)
1. Визуальная иерархия в крупных страницах местами “плоская”.
2. Нужна унификация severity-цветов/иконок на уровне дизайн-токенов.

## 4. Экранный аудит
### 4.1 Dispatcher
Источник: `apps/web/src/app/dispatcher/page.tsx`

Проблемы:
1. Много информации, но нет приоритизации “что горит прямо сейчас”.
2. Сложно понять, какой риск критичен и какое действие рекомендуется.

Решения:
1. Добавить блок `Critical Queue` над картой (SLA at risk, RTO risk, no-clearance).
2. Ввести `Recommended Actions` (кнопки one-click: “переназначить”, “отложить”, “split”).
3. Добавить боковую `Explain` панель по выбранному рейсу/ТС.

### 4.2 Logist
Источник: `apps/web/src/app/logist/page.tsx`

Проблемы:
1. Канбан хороший, но SLA/риск не встроен в карточку заявки.
2. Нет быстрого “бизнес-влияния” при смене статуса.

Решения:
1. Добавить на карточку: ETA risk, priority, “days in stage”.
2. Ввести полоску SLA по колонкам.
3. Сделать batch операции по выбранным карточкам.

### 4.3 Client Portal
Источник: `apps/web/src/app/client/page.tsx`

Проблемы:
1. Нет SLA прозрачности и причин отклонений.
2. Портал информативен, но слабо работает как инструмент доверия.

Решения:
1. Добавить `Live ETA` + интервал уверенности.
2. Добавить блок `Service Quality`: on-time %, avg delay, disputes.
3. Добавить таймлайн событий по заказу (с подтверждениями и документами).

## 5. Design System improvements
### 5.1 Tokens и семантика
1. Закрепить severity tokens: `critical`, `warning`, `ok`, `info`.
2. Убрать “случайные” цвета из экранов, использовать только семантические токены.
3. Добавить token для плотного режима (компактные отступы, шрифт, высоты строк).

### 5.2 Component states
Обязательные состояния для всех data-компонентов:
1. loading,
2. empty,
3. error + retry,
4. partial data,
5. stale data indicator.

### 5.3 Interaction patterns
1. Единый паттерн toast и error banners.
2. Единая паттернизация confirm modal для рискованных действий.
3. Hotkeys для диспетчера (`/` поиск, `A` assign, `R` replan).

## 6. Mobile UX guidance (Driver app)
1. Экран “Следующее действие” как главный.
2. Всегда видимый offline/sync статус.
3. Минимум текстового ввода, максимум выборов/шаблонов.
4. Явный конфликт-резолв после sync.
5. Крупные controls под работу в кабине (без мелких touch targets).

## 8. Метрики успеха
1. Время принятия диспетчерского решения: `-25%`.
2. Кол-во кликов на типовой сценарий назначения: `-30%`.
3. Ошибочные назначения/переназначения: `-20%`.
4. NPS клиентского портала: рост минимум на `+10`.
